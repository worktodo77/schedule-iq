BDI is a scored Report Card member (LI-06, weight 2 in Plan Stability). The
current demo series produces a plausible `0.0%`, but the focused synthetic
probes expose cases in which that same value means either true zero dilution
or an undefined denominator, and cases where a benign re-code becomes 50%
dilution.

## Scope and current behavior

Section 10.1 of `docs/ANALYTICS_PROPOSAL.md` says to take the current driving
path to the **selected milestone**, classify each step's working-day length as
baseline-original or post-baseline, and compute `100 * post-baseline step-days /
total step-days`. The engine is
`analytics/li_record.py::baseline_dilution_index`; it calls the shared
tool-of-record `driving_path` and emits LI-06 through `analytics/li_wiring.py`.
The active score curve is `src/scheduleiq/scorecard.yaml` LI-06:
`[[0,100], [25,70], [75,0]]`.

The existing tests cover range, one seeded added activity when it happens to be
on the selected path, and graceful handling of a single schedule. They do not
cover zero-length paths, UID re-coding, lag/type changes, explicit target
selection, or duration/provenance conventions.

## B1 - Zero-length path is scored as perfect dilution discipline (HIGH,
scored) - CONFIRMED

When `total_len <= 0`, the implementation sets `res.bdi_pct = 0.0` and adds
the reason `driving path has zero total length (all milestones)`
(`li_record.py:585-589`). A percentage whose denominator is zero is undefined,
not zero. The BDI wiring prefers the numeric value and therefore suppresses the
reason (`li_wiring.py:126-129`); LI-06 then maps `0.0` to 100 points.

**Probe.** A two-step all-milestone driving path, with both step lengths zero,
returns:

```
bdi_pct = 0.0
reason  = driving path has zero total length (all milestones)
steps   = [('A', 0.0), ('B', 0.0)]
```

Replacing a normal record's BDI with this result and running the actual wiring
produces `value=0.0` and the narrative `0.0% ... post-baseline`; the LI-06
curve consequently assigns 100. This is indistinguishable on the scorecard
from a measured path with genuinely zero post-baseline work.

**Ruling requested:** return `None`/N/A for a zero denominator and surface the
reason (or record a different explicit policy for all-milestone paths). Add a
seeded all-milestone fixture and pin the scorecard behavior. This changes a
scored result and requires the full governance quartet, CHANGELOG, and a
pinned-grade re-check.

## B2 - Code re-coding is reported as post-baseline work (Medium, scored) -
CONFIRMED

The baseline set is `{a.code}` and baseline edges are `(pred.code,
succ.code, type)` (`li_record.py:531-537`). Each latest path step is then
classified by `step.code in base_codes` (`li_record.py:543-551`). The model's
`uid` is the stable activity identity; `code` is an analyst-facing label.

**Probe.** Baseline and latest contain the same UID `a`, with baseline code
`A` and latest code `A-renamed`, on the same two-step path. The only change is
the benign re-code:

```
bdi_pct = 50.0
steps = [('A-renamed', False), ('B', True)]
decomposition = [('A-renamed', first_appeared='P (2025-02-01)')]
```

With the same code in both files the same path is 0% post-baseline. A pure
rename therefore changes the scored LI-06 number; code reuse can also make a
different activity appear baseline-original.

**Ruling requested:** use UID-first activity and relationship identity, with
code/name retained only for display. Rule whether a true UID replacement is a
new post-baseline activity. Add a seeded re-code test and require the full
quartet, CHANGELOG, and grade re-check.

## B3 - Modified relationships are inconsistently classified (Medium,
scored) - CONFIRMED

`base_edges` contains only `(pred.code, succ.code, rtype)`; it omits lag
(`li_record.py:532-537`). Consequently a lag-only modification remains
baseline-original even though section 10.1 explicitly includes an
**added/modified relationship** in the post-baseline class. The change register
does detect the modification as `kind="modified"`, but decomposition searches
only `lc.kind == "added"` (`li_record.py:570-579`). A relationship-type change
is therefore counted as post-baseline but is left with
`NOT FOUND - REVIEW` rather than the update in which it changed.

**Probes.** On a two-step path with both activities baseline-original:

```
FS lag 0h -> FS lag 8h:  logic_changes=[modified], bdi_pct=0.0,
                         both steps baseline_original=True
FS -> SS:                bdi_pct=50.0,
                         decomposition=[A->B, first_appeared='NOT FOUND - REVIEW']
```

The first result misses a number-changing logic modification; the second
reports dilution but cannot reproduce its first appearance from the
decomposition.

**Ruling requested:** define the relationship identity (pair, type, and lag or
the owner's chosen subset) and classify every ruled modification consistently.
Record the update for both `added` and `modified` changes. Add tests for lag,
type, and first-appearance output. Number-changing behavior requires the full
quartet, CHANGELOG, and grade re-check.

## B4 - No selected-milestone input reaches BDI (Medium, scored) - CONFIRMED

The specification says “current driving path to the selected milestone,” and
`driving_path` supports an optional `target_uid`. BDI's public function accepts
only `series_analysis` (`li_record.py:509`) and calls `driving_path(latest)` with
no target (`li_record.py:523`). The shared default chooses the latest-finishing
candidate, preferring incomplete/finish-milestone status. An analyst cannot
request a different contractual or delay milestone through BDI.

**Probe.** A spy replacement for `driving_path(sched, target_uid=None)` records
`target_args = [None]` when `baseline_dilution_index` runs. The BDI signature has
only `['series_analysis']`. The same latest schedule can have two terminal
paths with different baseline dilution, so the selected milestone changes the
number while the current API has no way to express that selection.

**Ruling requested:** either add an explicit target UID/code to the BDI API and
all callers, or amend section 10.1 to affirm the shared default target. Add a
two-target seeded fixture and score both ruled cases. This is a scored
methodology/API change and requires the full quartet, CHANGELOG, and grade
re-check.

## B5 - Length and baseline provenance conventions are implicit (Medium)

`_step_length_days` uses remaining duration when it is positive, otherwise the
original duration (`li_record.py:498-505`). Thus a live activity's contribution
shrinks as progress is recorded, while a completed activity retains its
original duration. The proposal and matrix say “working-day length” but do not
state this mixed basis, nor how missing/zero duration should be treated.

Separately, BDI takes `schedules[0]` as `baseline` and `schedules[-1]` as
`latest` (`li_record.py:518-520`). There is no explicit baseline marker or
validation that the first schedule is the contractual baseline; an input series
whose earliest file is an update is silently labelled as a baseline.

These are defensible choices, but they are methodology/provenance decisions,
not facts an implementation should silently choose. They materially affect
the denominator and the set of baseline elements.

**Ruling requested:** affirm the remaining-versus-original duration convention,
calendar fallback, and missing-duration policy; define how a baseline is
identified (positional first file or explicit provenance) and reject/flag a
series without one. Add disclosure fields and fixtures for an in-progress
step, a completed step, and ambiguous baseline provenance. Because the choices
change a scored value, use the full governance quartet and grade re-check.

## Lower-severity disclosure note

`BDIElement` has `code`, `kind`, and `detail`, but no `name`; the LI-06 wiring
therefore emits an empty Finding object name. This does not change the scalar
score, but it weakens the promised named-activity decomposition and should be
corrected or explicitly accepted when the BDI package is implemented.

## Proposed tests once rulings land

- **B1:** all-milestone path yields N/A/undefined, retains the reason, and is
  not awarded the 100-point zero-width curve.
- **B2:** UID-stable re-code remains baseline-original; UID replacement follows
  the ruling; display decomposition retains the latest code/name.
- **B3:** lag and type modifications follow the selected relationship identity;
  both receive a reproducible first-appearance update.
- **B4:** two selected milestones produce independently pinned BDI values; the
  default-target policy is explicit if no target is supplied.
- **B5:** remaining/original duration, missing duration, calendar fallback, and
  baseline provenance are pinned in seeded fixtures and disclosures.

## Governance package and rulings requested

All five findings can change LI-06's scored value. Each requires the full
governance quartet: matrix/methodology wording, implementation, seeded fixture,
tests, and a CHANGELOG note, followed by a pinned-demo grade re-check.

1. **B1 - zero denominator:** return N/A/undefined, or affirm an alternative
   all-milestone policy.
2. **B2 - identity:** UID-first matching; rule true UID replacement.
3. **B3 - logic modifications:** define pair/type/lag identity and classify
   `modified` changes with reproducible first-appearance labels.
4. **B4 - target:** add explicit selected-milestone input, or affirm the shared
   default target.
5. **B5 - basis/provenance:** affirm duration and calendar conventions and
   require explicit baseline provenance (or affirm positional first).
