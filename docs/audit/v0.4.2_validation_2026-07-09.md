# v0.4.2 independent validation — LOE/summary exclusion (LI kernel) + BWI B2 target

Date: 2026-07-09
Scope: independent re-verification of the v0.4.2 change (commit `2a8a9a1`)
against its pre-implementation parent (`e003bc4`), plus the two review rounds
that followed.  Method: two isolated git worktrees checked out at each commit,
the same probe scripts executed under both, output compared field-by-field.
Conclusions were re-derived from executed code — not read from commit messages
or prior audit notes.

**Verdict: SIGN OFF WITH NON-BLOCKING COMMENTS.**  The change is correct,
contained, and faithfully scoped.  The one residual (mixed-path PCI) is
pre-existing, disclosed, identical across both commits, and now pinned by a
regression test — a deferred methodology item, not a defect.

## 2. Change localization (corrected per review) — REPRODUCED

When LOE activities are introduced:

- **FCBI and CDI change by design** — they previously admitted the LOE as a
  weighted / dwelling entity and now exclude it.  (Series with a near-critical
  LOE: FCBI cumulative `[10,65]`→`[10,50]`, LOE burner `15.0` and recovery `2.5`
  removed; CDI drops the LOE leaderboard row and renormalizes.)
- **PCI may also change** where removal of a non-discrete path alters path
  concentration.  Reproduced directly: a real path `R1→R2→MS` plus a **pure-LOE**
  path `L1→L2→MS` gives PCI `0.555556` at `e003bc4` (the LOE path dilutes
  concentration) and `1.0` at `2a8a9a1` (the pure-LOE path is dropped).
- **RDI and BWI remain unchanged** — both already skipped LOE in their density
  loops (`if a.is_loe_or_summary or a.completed: continue`) and via the kernel
  RF map.

> Correction of record: an earlier draft of this validation stated "only FCBI
> and CDI change when LOE is present."  That was **wrong** — it held only for a
> probe that happened to use a *mixed* (kept) path, never a *pure*-LOE path.
> The committed docs (`matrix.yaml` LI-04, `ANALYTICS_PROPOSAL` §9.4) always
> described the PCI behavior correctly; the error was confined to the draft
> summary and is corrected here.

## 5. RF invariant (narrowed per review) — REPRODUCED

For **executable non-summary work activities**, no RF difference was found
between the two commits outside the approved LOE exclusions — and this holds
**by construction**: `float_paths()` enumeration is identical between commits;
the kernel filter removes only paths with *zero* discrete members; such a
removed path cannot contain any executable work activity, so no executable
activity's path-membership set changes and its RF is invariant (discrete members
`R1`/`R2` = `5.0`/`5.0` in every probe).

**Milestones are explicitly outside this invariant.**  A milestone's RF *can*
move: in the §2 pure-LOE case the target milestone `MS` moved `0.0 → 5.0`
because the path that gave it RF 0 (the pure-LOE chain) was dropped.  The
previously documented mixed-path PCI residual likewise remains unchanged.

## 6. PCI mixed-path residual — REPRODUCED, deferred (not a defect)

Smallest clean isolation — `A` (own float 30d) → `LOE` (2d) → `T` (5d), with a
competing real path `X` (50d) → `T`, so the LOE is the strictly-lowest-float
member of the driving mixed path:

```
                        e003bc4            2a8a9a1
driving path            [A, LOE, T]        [A, LOE, T]
  rel_float             2.0 (the LOE's)    2.0 (the LOE's)
rf_map[A]               2.0                2.0     ← A inherits the LOE's float
rf_map[LOE]             2.0                (absent) ← only difference
PCI (single update)     0.997429           0.997429
```

The LOE still drives the kept mixed path's `rel_float` (via the unchanged
`float_paths()`), and that float still propagates into the discrete member `A`'s
RF — **identically in both commits.**  v0.4.2 removes the LOE as a *weighted
entity* (gone from `rf_map`) and drops *pure* LOE/milestone paths, but does not
neutralize an LOE inside a kept mixed path.  This is a pre-existing, disclosed
limitation — **not a regression** — and is now pinned by
`test_pci_mixed_path_loe_residual_is_intentional`.  Any full neutralization must
be an LI-specific kernel/path calculation, never a change to the shared
`float_paths()` (§9.4 future-work).

## 7. Edge cases — REPRODUCED

all-LOE, all-milestone, LOE+milestone-only path, mixed discrete+LOE, empty
kernel (no relationships), no-milestone target, missing finish date, and
zero-remaining (all complete) all behave identically between commits except for
the two intended effects: pure LOE/milestone paths are dropped from the kernel,
and LOE keys are removed from the RF map.  The all-milestone case (no discrete
work) drops its sole path and PCI degrades to `None` — locked by
`test_pci_all_milestone_schedule_graceful`.
