## L1 — LI-02 scoring inverts the censored/died fraction (HIGH, scored) — CONFIRMED

**Implementation fact.** `scorecard.py::_li02_score` (L404-406):

```python
censored_frac = (ov.censored / ov.n) if ov.n else 1.0
if censored_frac < override.get("censored_pass_threshold", 0.10):
    return ov.median_months, 100.0, 0
```

`ov.censored` counts relationships **still alive** at the last update (did NOT
die).  The design spec (scorecard.yaml LI-02 rationale, verbatim from
REPORT_CARD_DESIGN.md): award 100 when the median was never reached **with
fewer than 10% of relationships having "died"** — "the network is so stable
there is not enough churn to even estimate a half-life."  "Fewer than 10% died"
is `died_frac = (n − censored)/n < 0.10`, i.e. `censored_frac > 0.90`.  The code
tests `censored_frac < 0.10` — the exact inverse.

**Probe (canonical stable fixture: 28 ties, 27 alive, 1 died, median not
reached).**

```
censored_frac = 0.964   died_frac = 0.036   threshold = 0.10
IMPLEMENTED  censored_frac < 0.10  -> 0.964 < 0.10 = False -> score 70
SPEC         died_frac    < 0.10  -> 0.036 < 0.10 = True  -> score 100
```

A maximally stable network is scored at the 70 placeholder instead of 100; the
100 branch fires only for a schedule where **>90% of logic has already died**
(median still unreached) — a case that essentially never occurs.  The 100 branch
is effectively dead code and the pass-mark placeholder is mis-applied to the
best schedules.

**Category / severity.** Implementation defect; HIGH; **scored** (LI-02, weight
2).  Fix: compare the **died** fraction to the threshold
(`(1 − censored_frac) < threshold`, or `died_frac < threshold`); consider
renaming `censored_pass_threshold` → `died_pass_threshold` to stop the
mislabel that produced the bug.  Number-changing → full governance quartet.

## L2 — Relationship death recorded one update too early (HIGH) — CONFIRMED

**Implementation fact.** `logic_half_life::to_variant` (L212) builds KM input as
`(last_alive_idx − birth_idx, censored)`.  For a **died** instance,
`last_alive_idx` is the last update where the tie was *still present*; the death
is observed at the *next* (first-absent) update.  So the event time is recorded
one update early.  For a censored instance (alive at the last update),
`last_alive_idx − birth_idx` is the correct censoring time — the bug is specific
to deaths.

**Probe.** Tie `A→B` present in U0, deleted in U1 (it survived one full update
interval):

```
key=(A,B) birth=0 last_alive=0 died=True  duration = 0   <- death at age 0
KM median = 0.0
```

A relationship observed alive and then deleted is recorded as dying at **age 0**,
which is impossible (it was alive when observed) and drags the KM curve below
0.5 at t=0.  Every died tie is under-aged by one interval, systematically
**understating** the logic half-life — and understatement makes the metric read
as *less* stable than reality, the adverse direction for a stability score.

**Category / severity.** Implementation defect; HIGH (biases the headline median
and feeds L1's scorer).  Fix: for died instances, event time =
`last_alive_idx − birth_idx + 1` (death observed at the first-absent update);
leave censored times unchanged.  The exact convention (right endpoint of the
interval-censored death) is a minor methodology sub-decision — recommend `+1`.
Number-changing → full governance quartet.

## L3 — Signatures keyed by activity CODE, not UID (Medium) — CONFIRMED

**Implementation fact.** `_relationship_signatures` (L133) keys each tie as
`(p.code, q.code, r.rtype.value)`.  When an activity is re-coded between updates
(a common, benign housekeeping edit; its UID is stable), every relationship
touching it changes key — the old signature is recorded as **died** and a new
one as **born**.

**Probe.** Activity uid `a` re-coded `A → A-RENAMED` between U0 and U1, logic
unchanged:

```
key=(A, B)         died=True    <- spurious death
key=(A-RENAMED, B) died=False   <- spurious birth
```

This inflates churn and shortens the half-life purely from renaming.  It is the
**same defect class the family already fixed for BWI (audit ruling B2:
persist the target by UID, resolve UID-then-code)**; LHL was not covered.  On a
re-coding-heavy series LI-02 (scored) is depressed with no real logic
instability.

**Category / severity.** Implementation defect (with a methodology facet — a
*type* change should still count as a death; a pure re-code should not); Medium;
scored on affected schedules.  Fix: key signatures by `(pred_uid, succ_uid,
rtype)`, mapping to codes only for display — mirroring B2.

## L4 — On/off-path ratio built from not-reached medians, unguarded (Medium) — CONFIRMED (Codex P2)

**Implementation fact.** `kaplan_meier` substitutes the last event time as a
lower-bound median when the curve never crosses 0.5, flagging
`median_reached=False` (L86-88).  `logic_half_life` then forms the on/off ratio
(L257-259) checking only that the medians are non-None/non-zero — **not**
`median_reached`:

```python
if (res.on_path and res.off_path and res.on_path.median_months is not None
        and res.off_path.median_months not in (None, 0)):
    res.on_off_ratio = res.on_path.median_months / res.off_path.median_months
```

**Probe.** on cohort median `1.0` (not reached), off cohort median `4.0` (not
reached) → `on_off_ratio = 0.25`, presented as an ordinary ratio.  A ratio of
two lower bounds is not itself a bound on the true ratio, so the number can
mislead in either direction.

**Category / severity.** Disclosure + minor correctness; Medium; **not scored**
(the scorer uses the overall median, not the ratio).  Fix: suppress the ratio
(or carry an explicit `ratio_reached` flag / caveat) when either cohort's median
is not reached; the narrative in `li_wiring.py` should qualify it.

## L5 — LOE/summary relationships enter the survival population (Low-Med) — CONFIRMED

**Implementation fact.** `_relationship_signatures` (L129-134) iterates all
`sched.relationships` with no discrete-work filter.  Reproduced: an `A→LOE` tie
is tracked as `('A','LOE','FS')`.  Logic churn on summary/hammock activities
therefore counts as "planning instability."

The family-wide LOE-exclusion ruling (2026-07-08, C1/X1) excluded LOE from every
kernel-consuming index but **did not reach LHL** (LHL does not use the kernel).
Whether a hammock's logic reshuffling is genuine planning instability or noise
is a methodology question.

**Category / severity.** Methodology decision; Low-Med; scored if ruled to
exclude.  Recommendation: exclude for family consistency (LHL measures *plan*
stability; summary logic is bookkeeping) — but this is the owner's call.

## L6 — Lower-severity items (disclose / rule)

- **Months conversion uses the mean update interval** (L214: `median_updates ×
  mean_interval / 30.44`).  On irregular cadences (e.g. 3 weeks then 3 months)
  the mean distorts the month figure.  Disclose the basis, or convert per-window.
- **On/off classification is fixed at birth** (docstring L175-176: "the update
  where the edge was first observed").  A tie that joins the driving path
  mid-life stays in the off-path cohort.  Defensible but undocumented.
- **Completed-activity ties never expire.**  Logic attached to finished work
  keeps "surviving" and censoring at the last update, potentially lengthening
  the median.  Consider whether ties into fully-completed successors should stop
  aging.  Methodology, disclose.
- **Disclosures.**  `LHLResult` carries only a `reason` string; FCBI/RDI/BWI/CDI
  now carry standing methodology disclosures (censoring %, exclusion default,
  units basis, signature definition).  LI-02 should carry equivalents (X-series).

## Proposed tests (lock behavior once ruled)

- **L1** — overall with `censored/n = 27/28`, median not reached → LI-02 score
  is **100** (not 70); and a churned case `died/n ≥ 10%`, not reached → 70.
- **L2** — a tie alive for exactly one update then deleted has event time **1**,
  not 0; a tie alive N updates then deleted has event time N; censored times
  unchanged; assert the resulting median exactly.
- **L3** — re-coding an activity (UID stable) does **not** register a death;
  a genuine relationship-type change **does**.
- **L4** — when either cohort's median is not reached, `on_off_ratio` is
  `None` (or flagged), not a bare number.
- **L5** — an LOE-attached relationship is excluded from the population once the
  ruling lands (a discrete tie with the same churn is retained).
