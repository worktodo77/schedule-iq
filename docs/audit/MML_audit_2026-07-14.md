LI-10 is informational and preliminary; it is not a current Report Card
category member. That does not make its output harmless: it is intended to
locate candidate measured-mile periods for expert review. The canonical demo
series returns five WBS rows, with activity-day fallback in all windows and
ratios of 0.0 for two rows; the probes below cover untested resource, identity,
event, and reporting behavior.

## Scope and current behavior

Section 10.5 of `docs/ANALYTICS_PROPOSAL.md` calls for productivity and its
dispersion per WBS/trade, scanning candidate windows for a **cleanest sustained
period** and impacted periods, and reporting the disruption contrast ratio
impacted productivity divided by clean-period productivity. The matrix labels
the metric preliminary and expects a per-top-level-WBS result.

`analytics/li_record.py::measured_mile_locator` creates one row per latest
top-level WBS and one window per consecutive schedule pair. It uses resource
actual-unit deltas when positive, otherwise activity-day fallback. The standard
LI wiring calls `run_li_record(sa)` with no events and then reduces all row
ratios to `min(ratios)`.

Existing tests check row/basis presence, ratio bounds, the preliminary caption,
and single-schedule degradation. They do not exercise mixed bases, re-coding,
dispersion/sustained selection, event injection, or the wiring representation.

## M1 - Cross-basis productivity ratios are dimensionally invalid (HIGH) -
CONFIRMED

For each window the implementation chooses resource productivity
`resource_delta / work_hours` when `resource_delta > 0`; otherwise it chooses
activity-day productivity `completed_duration_days / work_days`
(`li_record.py:865-885`). It then selects the maximum clean and minimum
impacted values across **all** valid windows and divides them
(`li_record.py:897-903`), without requiring a common `basis`.

Resource actual-units/hour and activity-days/working-day are different units;
their numeric ratio has no stable interpretation. The row retains each
window's basis, but the ratio computation ignores it.

**Probe.** A seeded WBS with three updates has a resource window at 1.25
actual-units/hour followed by an activity-day fallback window at 0.1667
activity-days/working-day:

```
windows = [('resource', 1.25), ('activity-day fallback', 0.1666667)]
clean basis = resource
impacted basis = activity-day fallback
reported ratio = 0.1333333
```

The 0.1333 value is presented as a disruption contrast even though numerator
and denominator are not commensurable. A negative/zero resource delta also
silently switches to fallback, masking a possible resource correction.

**Ruling requested:** require a common basis per comparison (or report
resource and activity-day contrasts separately), and define treatment of
negative/zero resource deltas. Add a mixed-basis fixture that must refuse or
separate the ratio. This is an output/methodology change and requires the full
governance quartet, CHANGELOG, and an expert re-check of the preliminary
surface.

## M2 - Stable-UID re-coding drops completed work (Medium) - CONFIRMED

The per-window activity lookup is `e_by_code = {a.code: a}` and resolves the
earlier activity with `e_by_code.get(la.code)` (`li_record.py:861-876`). A UID-
stable activity whose code changes is treated as absent from the earlier
window, so neither resource delta nor completed activity-days are credited.

**Probe.** One activity UID `a` is incomplete under code `A` in the earlier
schedule and completed under code `A-renamed` with 80 resource units later:

```
re-coded:  basis activity-day fallback, productivity 0.0, ratio None
same code: basis resource,              productivity 1.25, ratio 1.0
```

The metric loses real production solely because of a benign label change.
Code reuse can also attach a different earlier activity to the later one.

**Ruling requested:** match activities UID-first, retaining code/name for
display and an explicit replacement policy. Add a re-code fixture and test;
this changes the reported measured-mile candidate and needs the full quartet
and CHANGELOG even though LI-10 is not currently scored.

## M3 - Dispersion and sustained clean-period selection are absent (Medium)

The proposal asks for productivity **and its dispersion** and for the
“cleanest sustained period - stable, best productivity.” `MMLWindowRow` carries
only one productivity scalar; no variance, spread, sample count beyond the
activity count, or stability measure is calculated. `clean_window` is simply
the single maximum-productivity eligible window, while `impacted_window` is the
single minimum (`li_record.py:897-899`). The `spread_tight` boolean is a
15%-range heuristic used only to set `no_clean_mile`, not a dispersion output
or sustained-period selector.

Two schedules with one unusually high productivity window can therefore label
that one-off spike “clean” even when adjacent windows are unstable. The
current `no_clean_mile` flag does not preserve the dispersion that led to it.

**Ruling requested:** affirm the one-window max/min approximation, or define a
dispersion statistic and minimum sustained run/window selection. Keep the
preliminary label and record the expert's selection authority. Add a seeded
spike-vs-sustained fixture and expose the selected candidate evidence.

## M4 - Event exclusion is inert on the standard reporting path (Medium) -
CONFIRMED

`measured_mile_locator` accepts an optional `events` list and marks overlapping
windows (`li_record.py:837-839`, `887-888`). `run_li_record` can forward that
argument, but `analytics/li_wiring.py` invokes `run_li_record(sa)` without one;
there is no standard caller that supplies mapped disruption events.

**Probe.** Passing both canonical update windows as events directly marks 10
windows `excluded_by_event=True`; the standard `run_li_record(series)` path
marks 0 of 10 windows. Thus `clean_candidates` excludes events only when an
external caller has manually populated the optional argument. The documented
“no mapped events” clean-mile criterion is absent from ordinary report output.

**Ruling requested:** wire the authoritative event source into the standard
MML/report pipeline, or explicitly declare event exclusion opt-in and surface
that status. Add an integration fixture proving clean selection changes when a
window is event-excluded.

## M5 - LI wiring loses basis and per-WBS meaning (Low-Med) - CONFIRMED

Each `MMLWbsResult` stores windows, clean/impacted rows, ratio, and
`no_clean_mile`; it has no top-level `basis` field. The LI wiring nevertheless
formats `getattr(row, 'basis', '?')` (`li_wiring.py:181-187`), so every Finding
reports `basis ?` even when the underlying windows have a known basis. It also
sets the scalar LI-10 value to `min(ratios)` (`li_wiring.py:190`), despite the
matrix defining the metric per top-level WBS.

**Canonical fixture probe.** The actual standard wiring emits findings such as
`GEN: basis ?; ratio 0.00` and a narrative only for the strongest scalar
contrast. The row-level basis and which clean/impacted periods produced it are
not present in the Finding text.

**Ruling requested:** expose clean/impacted window basis and labels in the
decomposition, and affirm whether the scalar is a minimum, a display-only
rollup, or should be removed in favor of per-WBS output. This is a reporting
contract change; add an integration test and CHANGELOG note.

## Lower-severity disclosures

- Resource actual units from labor/material/equipment assignments are summed
  without a type normalization; the unit basis must be homogeneous or labelled.
- Activity-day fallback uses original duration for newly completed work, not
  actual elapsed duration; this is defensible but undocumented in section 10.5.
- WBS rows are determined from the latest schedule's root/top-level nodes; WBS
  re-parenting or a missing latest root can remove historical windows without a
  cross-update warning.

## Proposed tests once rulings land

- **M1:** mixed resource/fallback windows refuse or separate the ratio; negative
  resource corrections do not silently change basis.
- **M2:** UID-stable re-code preserves completed resource/activity-day output;
  UID replacement follows the explicit policy.
- **M3:** dispersion and sustained clean-period rule are pinned with a spike
  versus stable-window fixture.
- **M4:** standard report wiring passes mapped events and changes clean-window
  selection; opt-in behavior is explicit if retained.
- **M5:** findings expose WBS, basis, clean/impacted labels, and the selected
  rollup policy.

## Governance package and rulings requested

M1-M5 change the reported LI-10 evidence surface. Although LI-10 is not a
current Report Card member and is labelled preliminary, each accepted
number-changing or disclosure-contract ruling still requires the quartet:
matrix/methodology wording, implementation, seeded fixture, tests, and a
CHANGELOG note. The expert must confirm candidate period selection before
calling the metric suitable for a measured-mile analysis.

1. **M1 - basis:** separate/normalize resource and activity-day productivity;
   rule negative/zero resource deltas.
2. **M2 - identity:** UID-first matching and replacement policy.
3. **M3 - selection:** dispersion statistic and sustained-period rule.
4. **M4 - events:** wire authoritative events or affirm explicit opt-in.
5. **M5 - reporting:** expose basis/period evidence and rule the scalar rollup.
