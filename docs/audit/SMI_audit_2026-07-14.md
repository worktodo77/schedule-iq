## Scope and current behavior

Section 11.1 and the LI-11 matrix row define six weighted signals (TRD-05,
SET-01, LOG-10, DUR-03/-04, CAL-04, and statistical anomalies). Each signal is
scored from presence, saturating severity, and timing within 60 calendar days
before a claim/EOT submission; missing claim dates are supposed to be disclosed
and renormalized. The implementation is
`analytics/li_provocative.py::schedule_manipulation_indicator` and the
privileged surface is assembled by `analytics/li_wiring.py`.

The existing tests pin arithmetic, the demo decomposition, guardrail language,
and the absence of provocative indices from the standard card. They do not
exercise production event plumbing, timing for each signal, incomplete inputs,
historical DUR-03 scope, or large decompositions.

## S1 - Claim-event timing is dead on standard/internal callers (Medium) -
CONFIRMED

`schedule_manipulation_indicator` accepts `events` and extracts claim/EOT
dates, but the normal internal workbook path in `runner.py` calls
`li_provocative_results(sa, matrix, certificate=...)` without `events`. The
internal scorecard path does the same (`scorecard.py` RC5 helper). The standard
report intentionally omits SMI, so there is no other production caller that
supplies the event list.

**Probe.** Directly calling the metric with a claim event sets
`timing_available=True`; calling the normal no-event path leaves it false and
renormalizes every signal on presence/severity. The code supports timing in
theory, but a normal privileged run cannot use it.

**Ruling requested:** wire the authoritative event map into both privileged
callers, or explicitly make timing opt-in and print that status in the internal
workbook. Add an integration fixture with a claim date and a pre-claim edit.
This changes a privileged evidence output and requires the governance quartet
and CHANGELOG, even though the weight-0 grade remains unchanged.

## S2 - Timing attribution covers only two of six signal families (Medium) -
CONFIRMED

When events are supplied, the timing loop populates `change_dates` only for
`cs.actual_date_changes` (TRD-05) and `cs.calendar_def_changes` (CAL-04)
(`li_provocative.py` around lines 296-315). SET-01 settings flips, LOG-10
hollow-logic additions, DUR-03/DUR-04 edits, and STAT anomalies never receive
change dates. For a fired signal, their timing sub-score is therefore `None`,
which drops the 20% timing weight and renormalizes rather than recording a
within/outside-60-day result.

**Probe.** A fake series with one fired result for TRD-05, SET-01, LOG-10,
DUR-04, and CAL-04, plus a claim ten days after the edits, returns:

```
timing_available = True
TRD-05 = 100.0    SET-01 = None    LOG-10 = None
DUR    = None     CAL-04 = 100.0   STAT = None
```

This is not the published “each signal's edits” timing behavior. It makes a
claim-date-aware SMI depend on which two families happen to have date plumbing.

**Ruling requested:** add authoritative change dates for each signal (or rule
that timing is only meaningful for TRD-05/CAL-04 and revise the matrix). Add a
timing fixture per signal family and a decomposition disclosure.

## S3 - Missing inputs silently produce SMI 0 with no reason (Medium) -
CONFIRMED

`_grab` assigns count 0 when a requested series result is absent, and the
statistical helper also degrades to `(0, [])`. With two schedules but no
`sa.series_results`, `schedule_manipulation_indicator` returns a normal
`SmiResult` with `smi=0.0`, empty reason, and six zero-count signals. It does
not distinguish “no curation signals fired” from “the prerequisite series
checks were never computed.”

**Probe.** A two-schedule `SeriesAnalysis` with empty `series_results`, empty
assessments, and the statistical helper returning no result produces:

```
smi = 0.0
reason = ''
signals = all six count 0
```

On a privileged forensic surface this is false reassurance: the absence of
input evidence is not evidence of a clean record.

**Ruling requested:** require the named prerequisite results, return
NOT-COMPUTABLE/reason when they are absent, or explicitly document the
zero-default as a safe lower bound. Add an incomplete-input fixture and ensure
the internal workbook shows the distinction.

## S4 - DUR-03 is latest-file-only while the other signals span the series
(Medium)

The DUR signal adds the series DUR-04 count to only the most recent assessment's
DUR-03 value (`li_provocative.py` around lines 270-290). Earlier per-file
DUR-03 findings are discarded. TRD-05, SET-01, LOG-10, and CAL-04 are series
results, so their counts cover the whole update history; the mixed scope is not
stated in section 11.1 or the matrix formula.

**Probe.** Two assessments each carrying one DUR-03 finding, with DUR-04=0,
produce `DUR count = 1`, not 2. The latest assessment is the only one read.

**Ruling requested:** affirm latest-file scope, or aggregate all per-file
DUR-03 findings (with de-duplication rules) and disclose the scope. Add a
two-update fixture and pin the resulting signal severity.

## S5 - Full SMI decomposition is truncated (Low-Med) - CONFIRMED

`_grab` keeps `r.findings[:10]` for every signal. The privileged wiring then
uses only `sig.findings[0]` in the visible Finding text, although the embedded
decomposition retains the already-truncated list. The matrix row requires a
full decomposition of contributing signals and named, citable observations.

**Probe.** A TRD-05 result with 12 findings yields an SMI signal with exactly
10 findings; the LI-11 `MetricResult` Finding contains only the first one.
The remaining two observations are not available in the internal artifact.

**Ruling requested:** retain all findings (or publish an explicit evidence cap
with a count and separate detail artifact) and expose the signal's full list in
the privileged workbook. Add a >10-finding fixture and a no-loss assertion.

## Guardrail confirmation and lower-severity notes

- `SmiResult.privileged=True`, `surface="internal"`, the standing innocent
  explanations, and the sentence cap are present and tested.
- The six-signal arithmetic and no-claim renormalization are internally
  consistent with the current published formula; the issues above are input
  scope and evidence plumbing.
- Timing is calendar-day based and uses `0 <= claim - change <= 60`; this
  should remain explicitly labelled if timing is enabled.

## Proposed tests once rulings land

- **S1/S2:** internal runner and scorecard paths pass claim events; every fired
  signal receives a timing value or a documented not-applicable state.
- **S3:** missing series results return NOT COMPUTABLE/reason, not a clean 0.
- **S4:** historical DUR-03 aggregation/scope is pinned.
- **S5:** all contributing findings survive into the privileged decomposition
  and workbook.

scored surface would require a separate spec revision and governance ruling.

5. **S5 - evidence cap:** full decomposition versus an explicit capped artifact.
