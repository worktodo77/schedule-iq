## Scope and current behavior

Section 11.3 and LI-13 define ARR as each party's minimum attributed share
divided by its maximum share over every computable N4 variant. Zero-denominator
variants are skipped, parties with zero attribution in every variant are
excluded, and an N4 certificate with no responsibility overlay is NOT
COMPUTABLE. `analytics/li_provocative.py::_cert_view` (585-603) normalizes the
certificate and `attribution_robustness_ratio` (606-659) computes the ranges.

The scorecard entry is `N18`, `weight: 0`, `surface: internal`, and “needs the
N4 certificate.” Thus ARR's grade impact is none today; its privileged evidence
surface is the object under audit.

## A1 - A single variant can be labelled method-independent (Medium) - CONFIRMED

The implementation increments `n_variants` only for computable rows with a
positive total and then computes `min(shares) / max(shares)` without requiring
two or more variants. With one row, min and max are identical by construction.

**Probe.**

```text
certificate = {
  'overlay': True,
  'variants': [{'computable': True, 'per_party': {'Owner': 10,
                                                   'Contractor': 10}}]
}
ARR.n_variants = 1
Owner: min_share=0.5, max_share=0.5, arr=1.0
Contractor: min_share=0.5, max_share=0.5, arr=1.0
reason = ''
```

The output's “ARR 1.00 (method-independent)” interpretation is not supported
by a robustness comparison when the sweep has only one usable cell. This is a
methodology/spec gap, not a grade defect.

**Ruling requested:** require a minimum number of independent computable
variants (or emit a distinct “single-variant / not a robustness comparison”
state), and disclose the dimensions actually exercised.

## A2 - Excluded variant count and reasons disappear (Medium) - CONFIRMED

`attribution_robustness_ratio` skips `computable=False` rows and rows whose
per-party total is zero, but returns only the surviving count in `n_variants`.
It carries no total-variant count, skipped count, or certificate reasons into
`ArrResult`.

**Probe.** A three-row certificate with one usable row, one zero denominator,
and one non-computable row returns:

```text
ARR.n_variants = 1
Owner.n_variants = 1
Contractor.n_variants = 1
reason = ''
interpretation starts: Across 1 computable method variant(s), ...
```

The arithmetic follows the matrix's zero-denominator rule, but the privileged
artifact cannot show that two of three attempted cells were unavailable. This
is a disclosure gap and can make a thin sweep look stronger than its evidence
base.

**Ruling requested:** retain total/usable/skipped counts and the row-level
reasons (including zero-denominator), and include them in the internal
workbook/decomposition.

## A3 - Negative allocations are accepted as shares (Medium) - CONFIRMED

The code sums raw `per_party` values, rejects only a non-positive overall total,
and divides each raw value by that total. It does not validate non-negative
party allocations or clamp/report shares outside the [0,1] domain assumed by
the word “share.”

**Probe.**

```text
variants = [
  {'computable': True, 'per_party': {'Owner': -1, 'Contractor': 3}},
  {'computable': True, 'per_party': {'Owner': -2, 'Contractor': 4}},
]
Contractor shares = [1.5, 2.0]
Contractor ARR = 0.75
Owner = excluded (max share <= 0)
reason = ''
```

The output reports 150%-200% attribution and excludes the negative party as if
it had zero attribution. This is a domain-validation defect unless the N4
methodology explicitly permits signed movements; in that case the ARR spec
needs a signed-allocation rule rather than silently treating signed values as
shares.

**Ruling requested:** affirm non-negative delay allocations and reject/flag
invalid rows, or publish a signed-measure transformation and its disclosure.
Add a negative/zero allocation fixture.

## A4 - Responsibility overlay allocation is still code-keyed (High) - CONFIRMED

ARR itself consumes opaque `per_party` maps, but the N4 certificate producing
those maps is not UID-first. `robustness.py::_norm_responsibility` documents and
returns a code-to-party map (lines 354-370), and `_allocate_progress` and
`_allocate_revision` look up contributor/edit codes directly (648-675). A
UID-stable activity re-code therefore loses its party tag before ARR sees it.

**Probe.** A contributor with the same persistent UID but old code `OLD-CODE`
is allocated against a latest-schedule responsibility map keyed by its new code
`NEW-CODE`:

```text
resp_map = {'NEW-CODE': 'Owner'}
progress contributor = {'code': 'OLD-CODE', 'rd_change_workdays': 5.0}
allocated per_party = {'Unallocated': 5.0}
```

This is the exact R-ID failure mode: ARR can compare method variants whose party
shares changed because a code changed, not because responsibility changed. RSA
inherits the same certificate risk.

**Ruling requested:** port UID-first activity matching through N4 responsibility
allocation (retain code only as a narrow fallback/display field), add a recode
certificate fixture, and expose any fallback/unallocated identity loss.

## Guardrail confirmation and lower-severity notes

- ARR is privileged/internal, weight 0, and absent from the standard card.
- No certificate or no responsibility overlay correctly returns NOT COMPUTABLE.
- Zero-denominator rows are skipped as the matrix says; the finding is that the
  skip is not disclosed, not that the arithmetic skips it.
- ARR is a robustness screen, not an apportionment opinion; that guardrail is
  present in the matrix notes and certificate allocation note.

## Proposed tests once rulings land

- A1/A2: one usable row plus skipped rows returns an explicit coverage state and
  counts/reasons.
- A3: invalid signed allocations are rejected or transformed under a pinned
  signed rule.
- A4: same UID with changed code preserves the party allocation in N4 and ARR.

## Governance package and rulings requested

ARR is weight 0 and privileged, so none of these findings moves a Report Card
grade today. They do affect a discoverable internal robustness artifact. Any
accepted change requires the governance quartet: methodology/matrix wording,
implementation, seeded fixture/tests, and CHANGELOG note. Promotion to a scored
surface requires a separate spec revision.

1. **A1 - minimum sweep:** rule the minimum independent N4 coverage or a
   single-variant non-robust state.
2. **A2 - coverage disclosure:** publish attempted, usable, and skipped variant
   counts/reasons.
3. **A3 - allocation domain:** rule non-negative shares versus a signed measure.
4. **A4 - identity:** make the N4 responsibility overlay UID-first before ARR is
   relied upon.
