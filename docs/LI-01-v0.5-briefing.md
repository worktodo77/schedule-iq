**From:** ScheduleIQ implementation · **Date:** 2026-07-11 · **Re:** the governed
LI-01 revision, as implemented and shipped (rulings O1–O7,
[docs/rulings/LI-01-fcbi-v0.5.md](rulings/LI-01-fcbi-v0.5.md)).

This is the established briefing format (as-implemented definition → new worked

X/Y/Z worked example in the paper is **superseded**; use §(b) below as the new

## (a) Exact as-implemented v0.5 definition

Per window u, to one selected terminal completion milestone **m**, over the
**remaining-work population** (activities incomplete at the later update,
discrete, weight-resolvable):

1. **Target-specific distance (nonnegative).**  For activity i,
   `d_i = min over enumerated float paths containing i of (path margin to m −
   driving-path margin to m)`.  Driver d = 0; d ≥ 0 always; an activity on no
   enumerated path is *distance unresolved* → quarantine (never own-float).
2. **Weight.**  `w_i = 2^(−d_i / λ)`, λ = 5 working days ⇒ **w ∈ (0, 1]**.
3. **Consumption.**  `c_i = max(0, −ΔTF_i)` in reference working days
   (fixed reference 8 h/day; a Tier-1 tolerance of 0.5 h is applied to ΔTF *in
   hours* before conversion).  Recovery `= max(0, +ΔTF_i)` is tracked on a
   separate mirror.
4. **Primary outputs.**  `B = Σ c_i` (gross activity-day burn); `C = Σ(c_i·w_i) /
   Σ c_i` (burn-weighted mean proximity, **NOT APPLICABLE when B = 0**); optional
   `W = B·C`.  Recovery mirror **B⁻, C⁻** (never netted).
5. **Weight timing.**  Start-of-window `w(d_{i,u−1})` is primary; the
   **endpoint-timing sensitivity set** {start, end, min-endpoint} is exposed as
   a *sensitivity set*, not a band/bounds.
6. **Eligibility / governance.**  A contribution is eligible only with a directed
   path to m and no non-target governance of its late dates — **including
   governance propagated through the network** (constrained intermediate
   milestone).  Ineligible/unresolved → **quarantine subtotal**; report
   **eligible-burn coverage = eligible c / (eligible c + quarantined c)**.
7. **Severity, beside B and C (not in the kernel).**  `N_u = max(0, −F_m)` and
   `ΔN⁺ = max(0, N_u − N_{u−1})`.
8. **Segmentation & scale.**  Target-date / rebaseline / settings / calendar
   changes mark a **basis-change window** (excluded from the operational trend;
   requirement-induced margin change reported separately; cumulative restarts).
   Burn rate `= B / (working days)`; outputs carry actual data dates.

## (b) New worked example — the regression anchor (supersedes X/Y/Z)

One window to terminal completion milestone **COMP**, four workers on the
network `DRV→COMP`, `NEAR→COMP`, `GOV→GATE(MANDFIN)→COMP`:

| Activity | TF (start→end) | c | d (start) | w | eligible? | c·w |
|---|---|---|---|---|---|---|
| **DRV** (driver) | 0 → −4 | 4 | **0** | 1.00 | yes | 4.00 |
| **NEAR** | 5 → 2 | 3 | 5 | 0.50 | yes | 1.50 |
| **GOV** | 1 → −2 | 3 | (0) | — | **quarantined** (governed by GATE) | — |
| COMP (target m) | 0 → −4 | — | — | — | excluded (reference) | — |

Outputs (exact, reproduced by
`test_fcbi_worked_example_anchor_v05`):

- **B = 7.0** gross activity-days  (DRV 4 + NEAR 3)
- **C = 5.5 / 7 = 0.7857**  (W = 4·1 + 3·0.5 = 5.5, so **W = B·C = 5.5**)
- **eligible-burn coverage = 7 / (7 + 3) = 0.70**  (GOV's 3 quarantined)
- **N = 4.0**, **ΔN⁺ = 4.0**  (COMP fell to −4 working days of float)
- burn rate 0.28 activity-days/working-day over a 25-working-day window
- endpoint-timing sensitivity set (W): start 5.50 / end 5.31 / min 5.50

This single window exercises the d = 0 driver, a governed→quarantined
contribution, coverage < 100 %, and the N/ΔN⁺ severity output — the four things
the old X/Y/Z example could not show.

## (c) Before → after change log (v0.4.4 → v0.5), with probe numbers

Audited on the §P probe set (v0.4.4 "before" reproduced with hand-built
in-memory series; v0.5 "after" from the shipped implementation):

| Probe | v0.4.4 (audited) | v0.5 (shipped) |
|---|---|---|
| **P1** basis (X 0→−3, Y 5→9, Z 10→8) | Z off all paths → **own-TF fallback** (RF_Z = 8–10); a driver in negative float earns **w > 1** (up to 1.516, the "over-critical premium"); one scalar FCBI (3.5 no-logic / 6.5 with the milestone counted as a burner); **FCBI% 70–130 %** | d_X = **0** (driver), d_Z = **10**, every **w ≤ 1.0**; **B = 5, C = 0.7, W = 3.5**; FCBI% **retired** |
| **P2** cadence (d 10→0, c=10) | single min-RF-style scalar; cadence-dependent, undisclosed | timing set start/end/min = **2.5 / 10 / 10** (one window) vs **3.75 / 7.5 / 7.5** (two) — the min-RF supersession counterexample |
| **P3** identity | — | **W = B·C** on every window |
| **P4** opposite-direction | — | **W_min > both** endpoints (a sensitivity set, not bounds) |
| **P5** zero burn | FCBI = 0, no proximity reported | **C = NOT APPLICABLE** (labelled), never 0 |
| **P6** target pulled 10 d earlier | counted as ordinary burn, silently mixed into the trend | **basis-change window**; operational B = 0; **requirement-induced margin change = 10 d** reported separately; cumulative segmented |
| **P7** A→M(constrained)→completion | A **not** quarantined (counted at w ≈ 0.76); FCBI% 150 % | A **quarantined** (governance traced through M); coverage < 1 |
| **P8** split a burner | changes the weighted sum silently | changes **B** — the disclosed granularity property |
| **P9** unequal windows (21 d, ~90 d) | no normalization | **burn rate** present in every window |
| **P10** sub-precision hour jitter | any jitter → nonzero burn | jitter < tolerance → **B = 0** (Tier-1, in hours) |
| **P11** burns 15 d then completes | counted in burn (FCBI% 300 %) | **excluded from B**; in the **completion-omission** diagnostic with prior float + weight |

The two pathologies the reviewers flagged — the **w > 1 over-critical premium**
and the **absolute-TF fallback** — are both eliminated by ruling O1; the
**FCBI% > 100 %** readings they produced disappear with the retired ratio, their
real signal now carried explicitly by N / ΔN⁺.

## (d) Standing disclosures (carry these into the paper's guardrails)

1. **Within-network trend instrument only** — no cross-project or
   cross-granularity comparison; B is granularity-dependent (P8).
2. **B is a gross activity-day aggregate**, containing replicated path float —
   not "project float consumed" and not a stock.
3. **(B, C) does not cure** granularity or network-size dependence; it only makes
   network-size dominance visible.
4. **One target m in v1** — no summing across targets without an explicit
   allocation rule (double-counting); the analyst should select m explicitly.
5. **Coverage is disclosed, not thresholded** — no acceptable-coverage bar is
   invented; low coverage means much of the movement is off the eligible target
   basis, not that the schedule is sound.
6. **Governance is only as good as the tool-of-record data** (ADR-0004):
   unestablished governance → quarantine, so coverage is conservative.
7. **The endpoint-timing set is a sensitivity set, not bounds** — the
   within-window value can lie outside {start, end, min}.
8. **Tier-2 statistical noise treatment is not implemented** — Tier-1 absorbs
   precision only; do not read small non-zero B as necessarily material.
9. **Report Card scoring for LI-01 is provisional/ungraded** pending anchor
   recalibration on a real series; the index is reported informationally.
10. **Basis-change windows are real margin loss**, requirement-induced, not
    execution erosion — kept out of the operational-burn trend by design.

## (e) Questions for the paper author

1. **Target selection policy.**  v1 defaults m to the latest late-type-
   constrained finish milestone (the contractual deadline), else the last finish
   milestone.  Do you want the paper to mandate *analyst-selected* m with no
   silent default, or bless this default with the disclosure?
2. **λ and the near-critical intuition.**  λ = 5 wd is retained from v0.4.  With
   the distance now driver-relative (not absolute TF), do you want λ restated in
   the paper as "working days off the driving path" explicitly, and is 5 still
   the house value?
3. **Materiality rule (O7.6).**  We adopted `d difference > λ OR weight ratio >
   2` for record-network vs logic-only distance.  Confirm these thresholds for
   the paper, or supply the intended values.
4. **Reporting B vs W as the headline.**  We lead with B (gross) and treat W as a
   derived diagnostic.  For the paper's exhibits, should the cumulative curve be
   **B** (our choice) or **W**?  (We recommend B, with C annotated.)
5. **N / ΔN⁺ placement.**  Confirm the paper should present N and ΔN⁺ as a
   distinct severity strip beside the (B, C) exhibit, not folded into either.
6. **Coverage denominator.**  Coverage counts quarantined *burn* only.  Do you
   also want a *population* coverage (eligible activities / candidate activities)
   reported alongside, or is burn-coverage sufficient for the paper?
7. **Multiple-target roadmap.**  When v2 profiles several targets, what
   allocation rule do you want specified now to forbid double-counting later?
