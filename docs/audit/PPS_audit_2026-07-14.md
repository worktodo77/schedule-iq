## Executive summary

| # | Finding | Category | Severity | Grade impact | Evidence-surface impact |
|---|---|---|---|---|---|
| P1 | The proportionality criterion is a binary float-sign test, not a proportionality test | Defect / spec gap | High | None (weight 0) | Very different decelerations receive the same 100-point criterion |

The five published weights and neutral-50 rule are present. The defects are in
the evidence feeding two criteria and in the identity used by the upstream
§6.3 screen.

## Scope and current behavior

Section 11.4 and LI-14 define PPS for every §6.3 pacing candidate, with weights
float-at-start 0.25, contemporaneous awareness 0.20, proportionality 0.20,
reversibility 0.20, and re-acceleration 0.15. Missing evidence scores 50 and is
listed as neutral. `analytics/li_provocative.py::_pps_from_candidate`
(737-794) builds criteria; `pacing_plausibility` (797-820) gets candidates
from `analytics/pacing.py::pacing_candidates`.

The scorecard entry is `N19`, `weight: 0`, `surface: internal`; the standard
card omits it. Thus grade impact is none today, while the internal evidence
surface can be materially misleading.

## P1 - “Proportionality-to-float” ignores the deceleration magnitude (High) - CONFIRMED

When `float_at_start_days` is positive, `_pps_from_candidate` assigns
`proportionality = 100.0` and says only “deceleration occurred within ...
available float.” It never reads `cand.deceleration_measure` or compares the
deceleration with the available float. Any positive float therefore earns full
credit, whether the deceleration is a small fraction of float or consumes it.

**Probe.** Two candidates have the same five-day float and differ only in the
deceleration evidence:

```text
remaining duration +1h:    PPS 72.5, proportionality 100.0
remaining duration +1000h: PPS 72.5, proportionality 100.0
evidence for both: "deceleration occurred within 5.0 wd of available float"
```

This is not the published “was the deceleration proportionate to the float
available?” criterion. It is an implementation/spec defect and changes the
privileged PPS ranking even though the report-card grade remains unchanged.

**Ruling requested:** define the measurable numerator/denominator (including
units and treatment of negative/unknown float), compute the comparison, and
add small-versus-large deceleration fixtures.

## P2 - Re-acceleration is permanently neutral on the production path (High) - CONFIRMED

`_pps_from_candidate` reads `getattr(cand, "reacceleration", None)` (lines
780-787). `PacingCandidate` in `analytics/pacing.py` has no such field and
`pacing_candidates` never populates one. Consequently, every production PPS
instance receives a neutral re-acceleration criterion, regardless of what
happened after the parent delay resolved.

**Probe.** On the seeded three-update demo series:

```text
pacing candidates = 3
candidate has reacceleration attrs = [False, False, False]
PPS reacceleration criteria = [None, None, None]
neutral criteria for each = [..., 'reacceleration']
```

The neutral-50 behavior is correct for genuinely missing evidence, but this is
not missing input: the candidate model has no production field or detector for
the published criterion. Fifteen percent of every real PPS score is therefore
an untested placeholder.

**Ruling requested:** either implement a documented post-resolution
re-acceleration evidence field in the §6.3 candidate screen or remove/reweight
the criterion and revise the matrix. Add positive, negative, and unavailable
fixtures with explicit disclosures.

## P3 - Pacing candidate identity is code-keyed (High) - CONFIRMED

The upstream screen builds `e_by_code`/`l_by_code`, computes `matched_codes`,
looks up later activities by code, and emits `chain_codes`
(`pacing.py` lines 236-295). It does not carry the persistent UID from
`_noncritical_chains` into later-update matching. This violates the shipped R-ID
UID-first identity rule.

**Probe.** The seeded demo has three pacing candidates:

```text
before re-code: 3 candidates
  ... ['A1190'], ['A1080'], ['A1080']
same UID for A1080 is re-coded in the later schedule to A1080_RECODE
after re-code: 1 candidate (the two A1080 instances disappear)
```

Only the code changed; the activity UID and schedule changeset remained. PPS
therefore loses a pacing instance before scoring, and any code-keyed resource or
reversibility evidence is likewise at risk. This is an R-ID integration defect,
not a PPS weighting issue.

**Ruling requested:** match chain members UID-first across the pair, use a
narrow code-only fallback with disclosure, and add the re-code case to the
PPS/R-ID identity suite. Keep `chain_codes` as display labels only.

## Guardrail confirmation and lower-severity notes

- PPS is privileged/internal, capped as a preliminary triage score, and weight 0
  in `N19` / LI-14.
- The neutral-50 placeholder is disclosed per instance for fields that are
  genuinely absent, and the published weights sum to 1.0.
- The code-keyed identity issue is upstream of the arithmetic but directly
  changes which PPS instances exist.

## Proposed tests once rulings land

- P1: proportionality varies with a pinned deceleration/float ratio.
- P2: candidate-level re-acceleration evidence produces 0/50/100 with a
  named source, rather than an always-neutral criterion.
- P3: same UID/re-coded activity remains in the candidate and its chain label
  is disclosed.

## Governance package and rulings requested

PPS is weight 0 and privileged, so none of these findings moves a Report Card
grade today. They do affect the internal pacing evidence used for expert
triage. Accepted changes require the governance quartet: methodology/matrix
wording, implementation, seeded fixture/tests, and CHANGELOG note. Promotion
to a scored surface requires a separate spec revision.

1. **P1 - proportionality:** publish and implement a real deceleration-to-float
   comparison.
2. **P2 - re-acceleration:** supply evidence, or revise the criterion/weight.
3. **P3 - identity:** make pacing matching UID-first and disclose fallback.
