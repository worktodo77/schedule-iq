## Scope and current behavior

FRB is specified in `docs/ANALYTICS_PROPOSAL.md` section 9.3 and the LI-03 row
of `docs/METRIC_MATRIX.md` as the P10/P90 central-80% interval of actual minus
forecast finish, in working days, bucketed by forecast horizon. The engine is
`li_record.py::forecast_reliability_band` and `frb_apply_forward`; the series
surface is `analytics/li_wiring.py`; the Report Card uses the LI-03 curve in
`src/scheduleiq/scorecard.yaml`.

The existing probes (`tests/test_li_record.py`, four FRB tests) check populated
buckets, bucket statistics, low-n refusal, and empty-series handling. They do
not exercise the wiring field names, UID re-coding, bucket endpoints,
completed/LOE population, missing actuals, fractional forward dates, or a
reporting consumer for forward bands.

## F1 - LI-03 wiring silently collapses every non-zero band to zero (HIGH,
scored) - CONFIRMED

`FRBBucket` defines `bias_days`, `p10_days`, and `p90_days`
(`li_record.py:345-352`). `analytics/li_wiring.py:86-93` instead reads
`bias`, `p10`, and `p90` through `getattr(..., 0)`. Those attributes do not
exist, so the fallback is always zero. The value passed to `_res(..., "LI-03",
...)` is therefore zero whenever any bucket is populated.

**Probe 1 (isolated non-zero band).** A synthetic `FRBResult` with one bucket
`n=5`, `bias_days=4`, `p10_days=-2`, and `p90_days=13` has a true width of 15
working days. Running the actual `li_series_results` wiring produces:

```
true_width       15.0
wiring value     0
wiring narrative Empirical forecast error band (largest bucket): width 0 working days.
wiring finding   n=5, bias +0.0d, band P10 +0.0d .. P90 +0.0d
```

The LI-03 curve in `scorecard.yaml` is `[[0, 100], [20, 70], [60, 0]]`, so
the corrupted value receives 100 points. LI-03 is a weight-2 member of the
Forecast Credibility category (`scorecard.yaml:371-372`).

**Classification:** implementation defect.
**Scored impact:** the current curve awards 100 for the corrupted zero; the
intended non-zero width can score materially lower.

**Probe 2 (canonical demo series).** The real fixture returns two observations
and one populated bucket (`A1010`, P10=P90=47), yet the wired value and
narrative are still `0`.

**Control probe (central-80 interval and bias/dispersion split).** The engine's
bucket statistics on errors [0,1,2,3,4] return n=5, median bias 2.0,
P10=0.4, and P90=3.6, for width 3.2. Percentiles use linear interpolation;
the internal central-80 math and the separate median/dispersion values are
correct on this control. The defect is that the wiring reads different field
names and discards them.

**Ruling requested:** correct all three field names and add an integration
test asserting a non-zero width reaches both the LI-03 value and Finding text.
This is number-changing and requires the full governance quartet, a CHANGELOG
entry, and a pinned-grade re-check.

## F2 - Code matching loses outcomes after a benign re-code (Medium, scored) -
CONFIRMED

`forecast_reliability_band` builds `code_index` keyed by `a.code`
(`li_record.py:415`) and resolves later activities with
`code_index[j].get(a.code)` (`li_record.py:426`). Activity `uid` is the stable
identity in the canonical model; `code` is an analyst-facing label.

**Probe.** Two schedules contain the same activity UID `u1`. In the first its
code is `A`, forecast finish is 2025-01-08, and it is incomplete. In the later
schedule its code is changed to `A-renamed`, actual finish is 2025-01-09, and
it is complete:

```
UID-stable, code re-coded:  observations 0
same UID and same code:     observations 1, horizon 2.0d, error 1.0d
```

The current behavior loses a real forecast outcome (and code reuse can attach
the wrong outcome). This is the same identity class already ruled on for BWI
and LHL. The family-wide ruling is recorded in
`docs/plans/li_family_rulings_2026-07-14.md`; FRB's local fix site is its
`code_index` construction and later lookup. Match by UID, retaining
code/name only for display.

**Classification:** implementation defect.
**Scored impact:** a missing or cross-attached outcome changes the bucket
population and P10/P90 width.

**Ruling requested:** adopt UID-first matching, with an explicit ruling for a
true UID replacement (that should remain unobserved or be treated as a new
activity). Add a seeded re-code fixture and test. This changes a scored LI-03
number and requires the full quartet, CHANGELOG, and grade re-check.

## F3 - Horizon zero is observed but dropped from every bucket (Medium,
scored) - CONFIRMED

The bucket labels declare `0-30d`, but `_bucket_for` uses the strict lower
bound `lo < horizon_days <= hi` (`li_record.py:362-369`). The engine computes
`horizon = (forecast_finish - data_date).days` (`li_record.py:423`), so a same-
day forecast produces a valid `0.0` horizon that matches no bucket.

**Canonical fixture probe.** `forecast_reliability_band` returns:

```
observations: 2
MS-000: horizon 0.0d, error 65.0d       # observed, but unbucketed
A1010:  horizon 14.0d, error 47.0d       # bucket 0-30d
bucket counts: [('0-30d', 1), ('30-90d', 0), ('90-180d', 0), ('>180d', 0)]
_bucket_for(0): None
```

The complete endpoint probe is:

_bucket_for(-1) -> None
_bucket_for(0) -> None
_bucket_for(0.1) -> 0-30d
_bucket_for(30) -> 0-30d
_bucket_for(30.001) -> 30-90d
_bucket_for(90) -> 30-90d
_bucket_for(90.001) -> 90-180d
_bucket_for(180) -> 90-180d
_bucket_for(180.001) -> >180d

An isolated zero-day series has one observation, all four bucket counts zero,
and no forward bucket. Negative horizons are also silently outside the
defined buckets. Whether a same-day forecast is eligible is not stated in
section 9.3, but the `0-30d` label and the observed record make the current
silent loss unsafe.

The implementation also uses calendar-day `.days` truncation for horizon,
while the error interval is in working days. A forecast 30 calendar days plus
23 hours out is still bucketed as 30, and no calendar-versus-working horizon
or sub-day rounding rule is disclosed.

**Classification:** boundary implementation defect plus methodology gap.
**Scored impact:** changing inclusion or horizon basis changes bucket
population, P10/P90, and the LI-03 score.

**Ruling requested:** either include `0` in the first bucket (and state the
negative-horizon policy), or explicitly exclude same-day/negative forecasts
with a disclosed reason. Either choice changes population and band values,
so it requires the full quartet, CHANGELOG, and grade re-check.

## F4 - The promised forward band is not surfaced, and fractional shifts are
wrong (Medium)

Section 9.3 promises a forward completion-date band. `frb_apply_forward` is
defined (`li_record.py:447-466`) but has no production caller: the only
references are its definition and the low-n unit test. `li_wiring.py` emits
only a scalar width and bucket Finding; it never emits P10/P90 completion
dates. The live LI-03 narrative therefore cannot produce the documented
"schedule says ... P80 is ..." result.

An isolated n=5 bucket with P10=-2 and P90=3, forecast finish 2025-01-06,
and as-of 2025-01-01 returns forward dates 2025-01-02 and 2025-01-09 with
an empty reason. Missing dates return a reason, n<5 returns a low-sample
reason, and as_of equal to the forecast finish returns
"horizon 0d falls outside the defined buckets".

The helper used by that unconsumed API also advances one whole working day
before applying any fractional remainder. With a Monday 2025-01-06 00:00
start:

```
_add_workdays(start,  0.5) = 2025-01-07 04:00  # one full day plus 4h
_add_workdays(start, -0.5) = 2025-01-02 20:00  # one full day backward plus 4h
_add_workdays(start,  1.5) = 2025-01-08 04:00
```

For an 8-hour working-day convention, a half-day remainder must not add a
full extra working day. Percentile interpolation can produce fractional P10
or P90 values even though individual date errors are date-granular.

**Classification:** integration/implementation defect plus disclosure/spec
gap.
**Scored impact:** no current Report Card number moves because the API is
unused; the promised forward output is absent or wrong.

**Ruling requested:** decide whether the forward band is a required report
surface or a deliberately API-only helper; if required, wire it into the
series/report output and publish the time-of-day convention. Correct and test
signed fractional shifts. This does not currently move the Report Card
number, but it changes a promised LI-03 output and therefore still needs a
documented methodology/implementation/test/CHANGELOG package.

## F5 - Score basis conflicts with the published design (Medium, scored)

`docs/REPORT_CARD_DESIGN.md:109` says FRB is scored on band width **relative to
forecast horizon**. The active RC1 mapping in `src/scheduleiq/scorecard.yaml:457-460`
scores the raw largest-bucket width in working days with fixed points
`0 -> 100`, `20 -> 70`, `60 -> 0`; scorecard.py::_li02_score uses the
generic series-member curve (there is no FRB-specific normalization). A
10-day band at a 15-day
horizon and a 10-day band at a 180-day horizon therefore receive the same
score, contrary to the design sentence.

This is a methodology ownership question rather than an arithmetic defect:
absolute width may be the intended RC1 ruling, but the documents and the
scorecard currently disagree.

**Probe.** A 10-working-day band at a 15-day horizon is 66.7% of horizon,
while the same band at a 180-day horizon is 5.6%. The active absolute curve
gives both raw width 10 the same score (85 points), despite the design
sentence calling for relative width.

**Classification:** methodology/spec mismatch, not an arithmetic defect in
the interpolation curve.
**Scored impact:** selecting absolute versus relative width changes LI-03
scores and can reorder report-card results.

**Ruling requested:** affirm absolute working-day width and correct the design
text, or adopt a horizon-relative normalization and republish the curve. The
choice changes LI-03 scores and needs the full quartet, a CHANGELOG entry, and
a pinned-grade re-check.

## F6 - Completed and LOE populations are asymmetric (Medium, scored) -
CONFIRMED

The source population iterates u.incomplete_activities. That excludes
completed activities and excludes LOE/WBS/summary/hammock records through the
activity classification. A source forecast attached to an LOE therefore never
enters the bands. Conversely, after a source real task is forecast, the later
lookup checks only code, actual_finish, and later data date; it does not reject
a later LOE/type transition. A task that later becomes an LOE can therefore
enter the band.

**Probe.**

source LOE (incomplete), later completed:       observations 0
source Task (incomplete), later becomes LOE:    observations 1, error 1.0d
source Task already completed at source:        observations 0

This is a population rule, not just a display issue: the same activity-type
transition is included or excluded depending on which side of the forecast it
occurs. The open R-LOE ruling in the canonical handoff has not yet selected
the FRB policy.

**Classification:** spec gap / methodology ownership, with an implementation
predicate encoding the undocumented choice.
**Scored impact:** inclusion/exclusion changes sample counts and percentile
bands; a bucket can also cross the minimum n=5 refusal threshold.

**Ruling requested:** define whether FRB is real-activity-only at both
forecast and resolution, whether LOE-attached forecasts are eligible, and how
type transitions are handled. Add seeded source-LOE, task-to-LOE,
source-completed, and mixed-population fixtures. Number-changing rulings
require the full governance quartet, CHANGELOG entry, and pinned-grade
re-check.

## F7 - Missing or late actual finishes are dropped without censor disclosure
(Low-Medium, scored through population) - CONFIRMED

An observation is appended only when a later activity has actual_finish and
actual_finish <= later.data_date; the search then breaks at the first
qualifying update. A completed activity with absent actual_finish contributes
no observation. A finish present in a later file whose data_date is before
that finish is skipped, even if a subsequent file has a usable actual. The
result exposes only a global no-observation reason; it does not report
pending, late, missing, or censored forecast counts.

**Probe 1 (late then usable).** Forecast finish 2025-01-08; first later
snapshot has data date 2025-01-09 and actual finish 2025-01-10 (too late),
and the next snapshot has data date 2025-01-11 with the same actual:

observations 1, resolved at P(2025-01-11), error 2.0d

**Probe 2 (absent actual).** A later snapshot marks the activity completed but
leaves actual_finish=None:

observations 0
reason "no activity in the series both carried a forecast finish and later reported an actual finish"

The late case is eventually usable, but the missing case is indistinguishable
from no eligible forecast after aggregation. There is no denominator or
censoring disclosure for an audit consumer.

**Classification:** disclosure gap (and a spec gap if censored outcomes are
intended to be modeled).
**Scored impact:** current scores use only resolved outcomes; changing the
eligibility/censor policy changes n and the central-80 band, especially near
the minimum n=5.

**Ruling requested:** disclose counts for forecasted, eligible, unresolved,
late, and missing-actual records, and rule whether unresolved forecasts are
excluded, right-censored, or retained in a separate population. Add fixtures
for absent actuals and late-then-usable snapshots. Require the full quartet
and grade re-check if the scoring population changes.
