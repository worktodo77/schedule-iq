## L1 — Scoring censoring branch is inverted vs its own published rationale (HIGH) — CONFIRMED

**The published rule** (scorecard.yaml L445-454, self-described as the design
mapping "verbatim"):

> 100 if the Kaplan-Meier median survival is >= 12 months, **or the median was
> never reached with fewer than 10% of relationships having 'died'** (i.e. the
> network is so stable there is not enough churn to even estimate a
> half-life); … If the median was not reached but 10% or more of relationships
> **have already changed**, scored at the 70-point placeholder…

**Implementation fact** (`scorecard.py` L404-407):

```python
censored_frac = (ov.censored / ov.n) if ov.n else 1.0
if censored_frac < override.get("censored_pass_threshold", 0.10):
    return ov.median_months, 100.0, 0
return ov.median_months, override.get("not_reached_partial_score", 70.0), 0
```

`censored_frac` is the fraction **censored** (survivors). The design condition
is about the fraction that **died** (`1 − censored_frac`). The test is
inverted: 100 is granted when *fewer than 10% survived*, i.e. when *more than
90% died* — the opposite of "so stable there is not enough churn".

**Probes.**
- *Frozen network* (20 ties, 3 updates, zero changes): `censored_frac = 1.0` →
  **score 70**. Design: 0% died < 10% → 100.
- *1 of 20 ties dies* (5% died, median not reached): score **70**. Design: 100.
- *Unreachability of the 100 branch:* to fire it you need `median_reached ==
  False` **and** >90% deaths. But when ~90% of lifespans are events the curve
  crosses 0.5 (probe: 19/20 die → median reached). The branch is dead code in
  any realistic series; the intended stable-network reward can never be paid.
- *Demo fixture* (28 signatures, 1 death, 27 censored, median not reached):
  as-scored LI-02 member = **70**; under the design-intended condition
  (`deaths/n < 0.10` → 100) the member = **100**, Plan Stability 62.49 →
  67.94, series overall 52.21 → 53.30. Letter holds at D on this fixture
  (pinned-letter tests unaffected), but on a card nearer a boundary this is a
  letter-grade defect.

**Category / severity.** Implementation defect — the code contradicts the
mapping's own published rationale line, not merely an undocumented choice.
**HIGH**: it moves a scored member on precisely the schedules the design most
wants to reward (stable networks), and it silently disables the 100 branch.

**Secondary edge (same function) — promoted to its own finding in Rev 2.**
When the median *is* reached but `median_months is None` (no usable data
dates → no update→months conversion), the reached branch is skipped and the
censoring branch scores it. See **L10** for the reproduced consequence (a
maximal-churn series scoring 100); the two defects should be fixed together.

## L2 — "Median not reached" falls back to the LAST EVENT time; degenerate lower bound (Medium-High) — CONFIRMED

**Implementation fact** (`li_record.py` L86-88): when S(t) never crosses 0.5,

```python
if res.median is None and last_t is not None:
    res.median = last_t          # last observed EVENT time
    res.median_reached = False
```

and the docstring sells it as "the last observed event time is reported as a
conservative lower-bound estimate".

**Why the bound is degenerate.** When the median is not reached, S stays
above 0.5 through the **end of follow-up**, so the median exceeds the *longest
observed lifespan (censored included)* — the max follow-up time. The last
*event* time can be arbitrarily smaller than that.

**Probes.**
- 1 death at t=0 + 19 ties censored at t=5 → `median = 0.0`,
  `median_reached = False`. Reported: "at least **0.0** updates" — while every
  surviving tie has demonstrably lived 5 updates. The defensible bound is 5.
- **Live on the demo card**: the fixture's one death is a birth-window death
  (t=0), so the standard report narrative today reads
  *"Logic half-life 0.0 months (median not reached — conservative lower
  bound; 27/28 relationships censored)"* — a stability metric answering
  "at least zero" on the most stable series the fixture set contains. The
  card member shows `value=0.0` with no not-reached flag at all.
- **Ratio contamination:** `on_off_ratio` (L257-259) divides
  `median_months` values without checking `median_reached` — it can quietly
  divide a *reached* median by a *fallback lower bound* (or two bounds by each
  other), which is not a meaningful ratio of half-lives.

**Category / severity.** The fallback *choice* is a spec gap /
methodology decision (the spec says nothing about not-reached reporting), but
choosing the last **event** time is an implementation defect against the
docstring's own promise of a *conservative lower bound* — it is not the bound,
and in the common early-death case it is maximally misleading. Medium-High:
the score is not touched (only reached medians hit the piecewise curve), but
the headline value, narrative, and ratio are.

**Recommendation.** Report the lower bound as `max(duration)` over all
lifespans (the longest follow-up), keep `median_reached=False`, and have the
narrative say "not reached after N updates (~M months) of follow-up; k/n
censored". Suppress or flag the on/off ratio when either side is unreached
(labelled sentinel, per the FCBI item-2 precedent).

## L3 — Signatures keyed by activity CODE; a re-code kills the whole neighborhood (Medium) — CONFIRMED-BUT (Rev 2)

**Implementation facts.**
- `_relationship_signatures` (L124-135): key = `(pred.code, succ.code,
  rtype.value)`; lag is **not** in the key but is tracked as `lag_state` — a
  sorted tuple of all `round(lag_hours, 3)` values seen under that key.
- `_build_instances` (L150-160): an instance dies when the key disappears
  **or** `lag_state` changes; a lag change closes the old instance (event) and
  opens a new one at the same update. A **type** change (FS→SS) changes the
  key itself: old key death + new key birth. So the spec's "deleted or
  modified" is honored — **a lag change IS a modification** (death +
  rebirth), and so is a type change. This matches `compare.diff`'s notion of
  logic modification (diff.py L169-174 compares sorted `(rtype, lag)`
  signatures per code pair), so LHL and TRD-03 tell one consistent story.
- Relationships whose pred or succ UID resolves to no activity are skipped
  (L131-132) — dangling logic is out of population, silently.
- **Duplicate parallel relationships** (same pred/succ/type, different lags —
  rare but legal in P6) are merged into ONE signature whose `lag_state` is the
  tuple of lags; a change to either lag kills and re-births the merged
  signature (probe P1c confirms). One tie's edit reads as the death of the
  pair. Minor, disclose.

**The re-code probe.** UID held stable, `code` changes `A100 → A100R` at
update 1, no logic touched. Result: instance `('A100','B','FS')` dies (event,
duration 0), `('A100R','B','FS')` born — the series reports a logic death that
never happened. Overall median collapses to the 0.0-months fallback (L2
compounds it). An activity-ID renumbering pass between updates would read as
wholesale logic replacement and crater the scored median.

**Cross-check — the BWI B2 lesson.** B2 (target resolved by first-update
identity; re-code broke it) was ruled and fixed by moving to **UID** in
v0.4.2. But the trade is not one-sided here: task **codes** survive
re-exports in which a tool reassigns internal UIDs, and the sibling logic-churn
metric (TRD-03, via `compare.diff`) is code-keyed too — switching LHL to UID
alone would make LI-02 and TRD-03 disagree about whether a re-code is churn.

**Category — peer-review verdict (Rev 2): CONFIRMED-BUT.** The behavior is
real and reproduced, but it should **not** be framed as defect-adjacent: this
is a **methodology / disclosure decision**, and the current choice is
defensible on its merits — code keying aligns LHL with TRD-03 /
`compare.diff` (one consistent family answer to "was this logic changed?")
and is *more* robust than UID keying against UID churn on re-export, which is
the more common real-world hazard. **Recommendation (owner's call):** retain
code keying and disclose "a re-coded activity reads as logic death" in §9.2 +
matrix, unless the principal explicitly rules for (B) UID-keying — which
would then need to be a coordinated ruling with TRD-03 so LI-02 and the
logic-churn check keep telling one story.

## L4 — Survival population includes LOE/summary-attached ties and completed-work ties (Medium) — CONFIRMED-BUT (Rev 2)

**Implementation fact.** `_relationship_signatures` iterates
`sched.relationships` with **no** `is_loe_or_summary` and **no** `completed`
filter — every relationship whose endpoints resolve is in the survival
population. (Probe P8a: ties into/out of a Level-of-Effort activity appear as
ordinary population members.)

**LOE half.** The v0.4.2 family-wide ruling (C1/X1) excluded LOE/summary from
the kernel-consuming indices on the grounds that hammocks are not real work;
LHL was out of that ruling's scope (it does not consume the kernel), so the
question was never asked for LI-02: *is a hammock's logic churn "planning
instability"?* Hammock/LOE ties are bookkeeping (they span other work; P6
re-ties them when the spanned set changes) — their churn is mostly an artifact
of maintaining the hammock, and their stability is not evidence of a stable
plan. Whichever way it goes, it is currently an **accidental** inclusion, not
a decision.

**Completed half.** Logic attached to finished work is effectively immortal —
nobody edits the logic of completed activities, and the exporter keeps the
relationships — so every completed region contributes permanently-censored
ties that sit in the at-risk set at every event time and lift S(t).

**Probe (numbers).** 3 updates, 30d apart. Live ties: 6, of which 4 die in
the last window (duration 1), 2 censored. Completed-work ties: 6, never
changing.

| Population | n | censored | median | months | reached | LI-02 score |
|---|---|---|---|---|---|---|
| live ties only | 6 | 2 | 1.0 upd | 0.99 | **True** | **0.0** (piecewise at ~1 mo) |
| + 6 completed-work ties | 12 | 8 | 1.0 (fallback) | 0.99 | **False** | **70.0** (not-reached branch) |

Identical live-logic churn; the completed-work stock alone flips the median
from *reached* to *not reached* and moves the scored member from 0 to 70. On a
late-stage project (mostly completed work) LHL trends toward "not reached"
regardless of how violently the remaining live logic is being rewritten —
exactly the young-vs-old confound the scoring question (Q7/L1) worries about,
generated inside the population definition itself.

**Category — peer-review verdict (Rev 2): CONFIRMED-BUT.** The inclusion
behavior and its score effect are real (direction verified: *including*
completed ties **raises** the score via censoring inflation), but the
audit's censor-at-completion idea is a **methodology option to be ruled on,
not a fix to be applied**. The underlying question is genuinely the
principal's: is logic attached to completed work *valid historical stability
evidence* (the plan held all the way through execution — that IS stability),
or *dead-work ballast* that mechanically pads the censored mass on late-stage
projects? Both readings are coherent; they give different numbers.

**Options for the ruling** (no recommendation is privileged as "the fix"):
- **(a) LOE/summary-attached ties** — exclude (aligns with the family's
  "real work only" ruling) or consciously include with a disclosure.
- **(b) Completed-work ties** — (i) keep as-is (completed logic counts as
  surviving planning evidence; disclose the project-age sensitivity),
  (ii) censor a tie at the update both endpoints complete (its live history
  still counts; it stops accruing survival once unobservable as planning), or
  (iii) exclude ties born on already-completed work only. Option (ii) is the
  audit's proposal, presented as one candidate among these.

Whichever way each half is ruled, the choice and the censored fraction's
sensitivity to project age belong in §9.2 + disclosures (X1).

## L5 — Months conversion uses one global mean update interval (Medium) — CONFIRMED

**Implementation fact** (L204-215): lifespans are in **update counts**;
`mean_interval` = arithmetic mean of consecutive `data_date` gaps (in whole
days, `timedelta.days` truncation) over the post-exclusion series;
`median_months = median_updates × mean_interval / 30.44`. Pairs missing a
data date are skipped in the mean; if none is usable, `median_months` is
None (and the scoring falls into the L1 secondary edge).

**Probe — irregular cadence.** Ties survive a 21-day window, die during the
following 92-day window:

| Cadence | median (updates) | mean interval | reported months | actual survival window |
|---|---|---|---|---|
| 21d then 92d | 1.0 | 56.5d | **1.86** | 21–113d = 0.69–3.71 mo |
| uniform 92d | 1.0 | 92.0d | **3.02** | 92–184d = 3.02–6.04 mo |

The update-count KM is internally consistent, but the months figure inherits
whatever the *mean* cadence is: with a 3-week window followed by a 3-month
window, every "1 update" of survival is priced at 56.5 days regardless of
*which* windows the tie actually lived through. Two series with identical
real-time logic stability but different update cadences report different
half-lives; one series with a cadence change mid-record re-prices its own
past.

**Category.** Methodology decision + disclosure. The clean alternative is to
measure each lifespan **directly in days between data dates** (birth
`data_date` → death/censor `data_date`) and run KM in days — no conversion
step, immune to cadence irregularity, and it dissolves the L7 granularity
question into "which data date dates the death" (see L7; the two should be
ruled together). If update-count units are kept, disclose the mean-interval
basis, the 30.44 d/mo constant, and the caveat "compare only like-for-like
cadences" (the FCBI item-5 precedent). SPEC-CADENCE (D1) already scores
cadence regularity, so the series card does carry a co-signal — but LI-02's
own number should not silently depend on it.

## L6 — Baseline-pair exclusion: cohort-drop semantics are right, the 2-schedule edge misreports (Low-Med) — CONFIRMED

**Implementation facts** (L185-193).
- "Excluded" = the **baseline schedule is dropped from the cohort** (when ≥ 3
  schedules): deaths in the baseline→u1 window are never observed, and ties
  born in the baseline start their clock at update 1. This matches the
  docstring, and the probe confirms both halves (a tie deleted in window 1
  vanishes from `n` entirely; a baseline-born survivor's duration shortens by
  exactly 1). It is the stronger of the two readings of "first update pair
  excluded" — the matrix line ("baseline pair excluded by default") does not
  say which reading applies; the weaker reading (only *deaths* in window 1
  dropped, baseline births still clocked from 0) gives different numbers.
- **The edge:** with exactly **2 schedules** and `exclude_first_pair=True`,
  the condition `len(schedules) >= 3` fails and the baseline pair is used —
  but `LHLResult.exclude_first_pair` still reports **True**. Probe: a
  baseline-window deletion is scored as an event (`n=2`, median reached at 0)
  while the result claims the baseline pair was excluded. The output field
  states the *request*, not the *effective behavior*.
- Consequently the guard at L191-193 ("insufficient schedules remain after
  excluding the first update pair") is **dead code** — unreachable under the
  ≥ 3 condition. Harmless, but it documents an intent the code above it
  overrode.
- The toggle exists only as a Python parameter: `run_li_record` forwards it,
  but wiring and scoring always call with the default; nothing user-facing
  exposes or records it beyond the (sometimes wrong) result field.

**Category.** The stale flag is a (small) implementation defect —
`exclude_first_pair` should report what actually happened (or a separate
`first_pair_excluded_effective` field); the cohort-drop semantics and the
≥ 3 fallback are disclosure items for §9.2 + matrix.

## L7 — Deaths are dated at the LAST-SEEN update; the dying window is never counted (Medium) — CONFIRMED

**Implementation fact** (`_build_instances` L148-161 + `to_variant` L212):
`duration = last_alive_idx − birth_idx`. A tie present at u0 and u1 and gone
at u2 gets duration **1**, though it demonstrably survived the u0→u1 window
and died somewhere inside the u1→u2 window (true survival 1–2 update
intervals; probe: recorded 30d-equivalent for an actual 30–60d life).

The reality is interval-censored: the death happened in
`(last_seen, first_absent]`. The code imputes the interval's **left**
endpoint. Consequences:
- Every event time is understated by up to one full update interval; censored
  lifespans (counted to the last schedule) are not — so events are
  systematically shifted left *relative to* the censored mass, deflating the
  median.
- Any tie deleted — or lag-tweaked (L3 semantics) — within its first observed
  window is a **0-duration event**, dragging S(t) down at t=0. This is what
  puts "0.0 months" on the demo card (L2 shows the same event through the
  fallback lens).
- On a 3-update series (2 usable windows, the current fixture reality) the
  only possible event times are 0 and 1 — the metric's resolution is one
  window, and the left-point choice throws half of it away.

**Category.** Methodology decision + disclosure. Defensible options: left
point (current — most conservative, "survived k full windows"), right point
(`idx − birth` — "died by window k"), or midpoint (standard for
interval-censored data). **Recommendation:** midpoint, ruled **jointly with
L5** — "lifespan in days from birth data-date to the midpoint of the
disappearance window; censored at the last data-date" is one coherent
re-specification that resolves L5 and L7 in a single formula change. If the
left-point convention is kept, state it in §9.2 ("a tie deleted in its first
window has lifespan 0") so nobody reads the median as "time until deletion".

## L8 — On/off-path split: birth-only classification, silent drops, undocumented ratio direction (Medium) — CONFIRMED

**Implementation facts** (L222-259).
- An instance is classified by whether its `(pred_code, succ_code, rtype)`
  key is in the **driving path of the schedule at its birth index** —
  membership at birth, once, forever. The driving path is the single
  min-float backward walk (`analytics.paths.driving_path`), not a
  near-critical band.
- Probe: chain A→B→C driving at u0, tie X→C floaty at u0; floats flip at
  u1/u2 so X→C becomes *the* driving edge for the rest of the series. LHL
  buckets: on-path = {(A,B),(B,C)}, off-path = {(X,C)}. The tie that carries
  the driving path for two of three updates is counted as off-path logic.
  (Note this interacts with L3/L1's lag semantics: a lag change re-births the
  instance, which *re-classifies* it at the new birth — so a tie can migrate
  buckets only by being modified.)
- **Silent drops:** when `driving_path()` cannot resolve for an instance's
  birth update, the instance is skipped from **both** buckets (L246-247).
  Only when *every* birth update lacks a path is a reason set. Probe: 4
  instances overall, on+off = 3, one silently missing, `reason=''` — the
  split's population quietly ≠ the overall population, undisclosed.
- **Ratio:** `on_off_ratio = on_months / off_months` (guarded against None/0
  denominators). Direction — **ratio > 1 means driving-path logic is MORE
  stable** — is documented nowhere (§9.2 says only "driving-path logic should
  be MORE stable than average"); the wiring narrative prints a bare "On/off-
  path ratio 0.43" a reader cannot orient. And per L2, either side may be an
  unreached lower bound with no flag on the ratio.
- Degenerate cases verified: no driving path anywhere → overall-only with an
  explanatory reason (good); all ties on-path → `off_path=None`, ratio None
  (acceptable); the split is narrative-only (not scored), capping severity.

**Category.** Spec gap (birth-only is a real convention choice §9.2 never
makes) + methodology decision + disclosure. **This is a metric-intent
ruling, not an obvious correction** (Rev 2 revision): the candidate bases
measure different things, and none is simply "right" —
- **(A) birth-only** (current): "how stable is logic *as constituted* on/off
  the driving path when it was created?" Cheapest; mislabels path-joining
  ties like the probe's X→C.
- **(B) membership at any point in life:** "how stable is logic that *ever
  became* driving?" Catches the path-joiner, but a tie that brushed the
  driving path for one update is permanently "on", which overstates the
  driving bucket on path-churning schedules — arguably the very schedules
  this split is for.
- **(C) per-update classification / dwell fraction:** the most faithful, and
  the most machinery.

The audit's earlier lean toward (B) is withdrawn to neutral; the principal
should pick the intent. Independently of that ruling (these stand under any
basis): disclose the drop count when some birth updates lack a resolvable
path, state the ratio direction in §9.2/matrix/narrative, and suppress or
flag the ratio when either median is unreached (with L2).

## L9 — Data-date validation gap: out-of-order/same-day/missing dates convert silently (Medium) — NEW in Rev 2, CONFIRMED

**Implementation fact** (`li_record.py` L204-207, L214-215): the mean update
interval is a plain mean of `(b.data_date − a.data_date).days` over
consecutive pairs where both dates exist; there is no sign, zero, or
plausibility check before it scales the median into months.

**Probes** (4 ties, 2 dying at u2, so a reached ~1-update median exists):

| Data dates | mean interval | median_months | LI-02 score |
|---|---|---|---|
| out-of-order (DD runs backwards) | **−30.0** | **−0.99 months** | 0.0 (negative value fed to the curve) |
| all same-day | 0.0 | None (`mean_interval` falsy) | 70.0 (via L10 fall-through) |
| all missing | None | None | 70.0 (via L10 fall-through) |
| baseline DD missing only | 30.0 (surviving pair) | 0.99 | 0.0 — reasonable degradation |

A negative half-life is nonsense on its face and reaches the card value and
narrative without any warning or `reason`; the None cases are silent too and
route into the L10 fall-through. (The partial-missing case degrades sanely —
the mean is taken over the pairs that have dates — worth keeping and
stating.)

**Category / severity.** Implementation defect (missing input validation) +
disclosure; Medium — out-of-order data dates are exactly the kind of thing a
distressed real-world series produces, and TRD/intake checks flagging them
elsewhere does not excuse LI-02 emitting a negative month figure. Guard
(non-positive or missing intervals → labelled sentinel + reason, months
withheld) once L5's units ruling lands — a per-instance day basis needs the
same validation at the lifespan level.

## L10 — Reached median in updates + no usable months = scored by the censoring branch (HIGH) — NEW in Rev 2, CONFIRMED

**Implementation fact** (`scorecard.py` L402-407): the reached-median branch
requires `ov.median_reached and ov.median_months is not None`. When the
median IS reached in update units but the months conversion is unavailable
(L9's None cases — missing or same-day data dates), the function falls
through to the **censoring** branch, which was never meant to see a
reached-median series at all.

**Probe.** 20 ties, 19 die at update 1 — a maximally churning network whose
median is reached (1.0 update):

| Data dates | median reached? | censored/n | LI-02 score |
|---|---|---|---|
| present (30d cadence) | yes, 0.99 months | 1/20 | **0.0** (curve, correctly) |
| missing | yes (updates), months=None | 1/20 | **100.0** |

With months unavailable, `censored_frac = 0.05 < 0.10` fires the **inverted
L1 branch** and awards a perfect score to the least stable series the metric
can express. The same schedule swings 0 ↔ 100 on nothing but data-date
availability.

**Category / severity.** Implementation defect, **HIGH** — it compounds L1
(the inverted branch is what makes the fall-through catastrophic rather than
merely wrong-bucket) and moves a scored member by the full scale. Fix
belongs with L1 in one ruling: the censoring branch must be reachable only
when the median is genuinely not reached, and a reached-but-unconvertible
median needs an explicit path (score on update units with a disclosed
assumed cadence, or return ungradeable/N-A with a reason — owner's choice,
the L5/L9 rulings feed it).

## X2 — Duplicate-tie multiplicity is collapsed; deleting one duplicate reads as death + rebirth (Low) — NEW in Rev 2, CONFIRMED

**Implementation fact** (`_relationship_signatures` L124-135): parallel
duplicate relationships — same (pred code, succ code, type), different lags,
rare but legal in P6 — are merged into ONE signature whose `lag_state` is the
sorted tuple of `round(lag_hours, 3)` values.

**Probe.** u0 carries `A→B FS` twice (lags 0h and 16h); u1 deletes the 16h
duplicate. `lag_state` goes `(0.0, 16.0)` → `(0.0,)`: the merged instance
dies (event, duration 0) and a new instance is born — the surviving lag-0
tie, which existed unchanged the whole time, is credited with **no**
continuous survival. Deleting one duplicate is indistinguishable from
modifying the pair.

**Category.** Disclosure (rare-input semantics), Low; marginal score effect
on real files. The behavior is a reasonable consequence of the merge design
(and *some* death signal on a duplicate edit is arguably right); it just
needs a sentence in the signature-definition disclosure (X1 item 1) so a
re-runner isn't surprised. If the principal wants partial survival instead,
that is a signature-definition change (per-duplicate instances keyed with a
multiplicity index) and belongs with the L3 ruling.
