IL is a scored Report Card member (LI-08, weight 2 in Float & Progress
Management). The active special curve is `0 updates -> 100`, `2 -> 70`, `6 ->
0`, with an unresolved-only fallback of 20 points. The canonical fixture
returns one event with latency 1 update / 91 calendar days; the focused probes
below expose cases the existing tests do not cover.

## Scope and current behavior

Section 10.3 of `docs/ANALYTICS_PROPOSAL.md` defines IL as median updates (and
calendar days) from a chain first showing negative float to its first
responsive edit, with unresolved events counted separately. Responsive edits
include logic resequencing, duration compression, calendar upgrades, resource
additions, and recovery re-baselining. The engine is
`analytics/li_record.py::intervention_latency`; it consumes the code-based
`compare()` change register and is scored by `scorecard.py::_li08_score`.

The existing tests cover one seeded emergence, graceful no-event behavior, and
the bundle. They do not cover same-pair responses, mixed resolved/unresolved
events, re-coding, deleted logic, remaining-duration edits, or chain/LOE
population rules.

## I1 - Same-update responses are invisible and zero latency cannot be scored
(HIGH, scored) - CONFIRMED

The emergence is found in changeset `i`, but the response scan starts at
`for j in range(i + 1, len(changesets))` (`li_record.py:715`). A duration or
logic edit in the same update pair that caused the float transition is never
examined. The smallest emitted `il_updates` is therefore 1 (`li_record.py:721`)
even though the published LI-08 curve explicitly awards 100 at 0 updates.

**Probe.** One activity moves from TF 0 to -1 working day in a pair and its
original duration simultaneously decreases from 80h to 40h:

```
change register: float_deltas={'A': -1.0}, duration decrease A: 80h -> 40h
_response_in(current_pair, {'A'}): duration decrease on A: 80h -> 40h
IL event: il_updates=None, unresolved=True, median_il_updates=None
```

The response is objectively present in the emergence pair, but IL reports
“unresolved.” A response in the next pair yields 1 update, never 0.

**Ruling requested:** define same-pair response as 0 updates (recommended, and
consistent with the published curve), or explicitly exclude it and republish
the curve. Add a seeded same-pair fixture and pin the score. This is a scored
number change requiring the full governance quartet, CHANGELOG, and grade
re-check.

## I2 - Mixed unresolved events do not affect the Report Card score (HIGH,
scored) - CONFIRMED

`ILResult` correctly counts unresolved events, but `_li08_score` returns the
piecewise score as soon as `median_il_updates` is non-None
(`scorecard.py:417-428`). The median is computed only from resolved events
(`li_record.py:731-738`), so unresolved events are ignored whenever at least
one event resolved. The fixed 20-point branch applies only when *all* events
are unresolved.

**Probe.** Two chains emerge together; chain A receives a duration response in
the next pair, chain C receives no response:

```
events: A -> il_updates=1, unresolved=False
        C -> il_updates=None, unresolved=True
median_il_updates = 1
unresolved_count = 1
```

The special scorer therefore evaluates only `1` against `[0,100],[2,70],[6,0]`
(85 points by linear interpolation) despite one chain never receiving a
response. The wiring narrative says “1 unresolved,” but the grade does not
reflect it.

**Ruling requested:** decide whether unresolved events should cap/penalize a
mixed score, be reported outside the grade, or remain resolved-only. The
current implementation silently chooses resolved-only; record the principal's
choice and add a mixed-event score fixture. Any scoring change requires the
full quartet, CHANGELOG, and pinned-grade re-check.

## I3 - Re-coding loses both emergence and response evidence (Medium, scored) -
CONFIRMED

`compare()` matches activities by `code` (`compare/diff.py:90-97`), and IL then
builds earlier/later maps by code (`li_record.py:693-699`). A stable UID that is
re-coded is treated as deletion plus addition, so it creates no float delta.
The response helper also resolves resources by code (`li_record.py:669-675`).

**Probe.** UID `a` moves from TF 0 to TF -1 while code changes `A` to
`A-renamed`:

```
float_deltas = {}
deleted = ['A']; added = ['A-renamed']
IL events = 0
reason = no activity transitioned from non-negative to negative total float
```

The metric misses a genuine emergence and every later edit touching the same
UID under its new code. This is the same identity class already surfaced in
LHL, FRB, and BDI audits.

**Ruling requested:** use UID-first matching, with code/name fallback only for
display or explicitly unresolvable exports; rule true UID replacement as a new
activity. Add re-code emergence/response fixtures. This changes a scored
number and requires the full quartet, CHANGELOG, and grade re-check.

## I4 - Valid response types are omitted (Medium, scored) - CONFIRMED

`_response_in` accepts only logic `added`/`modified`, duration changes,
calendar changes, and budget-unit increases (`li_record.py:650-678`). Two
implementation gaps follow:

1. A logic **deletion** is a plausible resequencing response but is not
   accepted; only `added` and `modified` are scanned.
2. `compare()` records only changes to `original_duration_hours`
   (`compare/diff.py:100-108`). A common recovery action that reduces
   `remaining_duration_hours` creates no `duration_changes` entry and is
   invisible to IL.

**Probes.** For a negative chain A/B, a later relationship deletion yields:

```
logic_changes = [('deleted', 'A', 'B')]
_response_in(...) = ''
IL event remains unresolved
```

For a stable-code activity whose remaining duration changes 80h -> 40h while
original duration remains 80h:

```
duration_changes = []
_response_in(...) = ''
```

Both cases bias latency upward or turn a resolved event into unresolved.

**Ruling requested:** define which logic and duration edits count as responsive
(including deletions and remaining-duration compression if intended), then
extend the change register and response detector together. Add one fixture per
response type. Number-changing behavior requires the full quartet, CHANGELOG,
and grade re-check.

## I5 - Chain population and LOE/summary handling are silent methodology
choices (Medium)

Emergence is defined as the set of activities whose own float crosses from
non-negative to negative. The later relationship graph is then restricted to
edges whose **both endpoints are in that set** (`li_record.py:695-711`). A
positive-float intermediate therefore splits a connected A -> B -> C path into
two events when A and C cross negative. The implementation also does not filter
LOE, WBS-summary, hammock, or summary activities before creating events.

**Probes.** With A -> B -> C and only A/C crossing negative:

```
events = [['A'], ['C']]       # B is positive, so it cannot connect them
```

When an LOE activity A crosses negative, it is emitted as an ordinary event:

```
events = [['A']]              # no discrete-work filter
```

Section 10.3 says “chain” and lists path/FAIL emergence but does not say
whether the chain is the induced negative-float subgraph, a full connected
logic chain through non-negative nodes, or whether LOE/summary rows are in the
population. These choices change event counts, unresolved counts, and medians;
they are not safe to infer from the current code.

**Ruling requested:** affirm the induced-negative-node basis or define a full
chain traversal, and rule LOE/summary inclusion. Add fixtures and standing
disclosures. This is a methodology decision; if the score population changes,
use the full quartet, CHANGELOG, and grade re-check.

## Lower-severity disclosures

- `il_days` is calendar-day difference between data dates, while float and
  duration inputs are calendar-specific/working-day quantities; the proposal
  says calendar days, but the mixed basis should be surfaced in the result.
- A resolved event can have `median_il_updates` but no `median_il_days` when
  data dates are missing; `li_wiring.py:154` then prints `None days`.
- Any calendar change counts as a response, including a downgrade; resource
  response is inferred only from increased budget units, not assignment
  additions with unchanged budget units.

## Proposed tests once rulings land

- **I1:** same-pair response produces `il_updates=0`; later response still
  produces 1; score curve pins both.
- **I2:** mixed resolved/unresolved events pin the selected scoring policy.
- **I3:** UID-stable re-code preserves emergence and response; UID replacement
  follows the ruling.

## Governance package and rulings requested

I1-I4 can change the scored LI-08 number; I5 can change its event population
and score. Each accepted number-changing ruling requires the full governance
quartet: matrix/methodology wording, implementation, seeded fixture, tests,
and a CHANGELOG note, followed by a pinned-demo grade re-check.

1. **I1 - same-pair response:** adopt 0-update latency or explicitly exclude it.
2. **I2 - mixed unresolved:** choose a scoring treatment for unresolved events
   when resolved events also exist.
3. **I3 - identity:** UID-first matching and true replacement policy.
4. **I4 - response taxonomy:** include/exclude deleted logic and remaining-
   duration compression; update the change register as needed.
5. **I5 - population:** define chain connectivity and LOE/summary inclusion.
