# LI-10 MML — Wave-4 governed revision (ruling Q-F: full package)

**Status:** accepted · **Date:** 2026-07-12 · **Principal:** Alex Bachowski
(Wave-4 triage on the LI-02..LI-10 audit, question Q-F — "Full package") ·
**Classification: number-changing** (informational metric — no Report Card

names in forensic output move).

## Rulings

1. **Basis segregation (audit MML-1, rubric A2/A3 — the top MML finding).**
   Each trade's clean/impacted comparison and ratio are computed within ONE
   basis: resource units consumed per working hour ONLY when every
   data-bearing window of the trade has resource movement; otherwise every
   window uses the activity-day basis (planned days completed per working
   day).  A resource window is never compared against an activity-day
   window.  The prior per-window auto-selection divided across the two
   bases (audit probe: "ratio 0.4" from 1.25 units/h ÷ 0.5 act-d/d — a
   dimensionally meaningless "disruption contrast" presented per trade).
2. **Sustained clean mile (audit MML-2, spec conformance).**  §10.5 defines
   the clean mile as "the cleanest sustained period — stable, best
   productivity, no mapped events"; the implementation took the single
   max-productivity window (a one-window ramp-up/closeout spike could become
   the measured mile).  The clean period is now the best MEAN over
   `MML_RUN_K = 2` consecutive, valid, non-event windows whose spread
   (max−min) is within `MML_RUN_SPREAD = 25%` of the run mean — both
   **recorded conventions**, not calibrated constants (per the ruling, k and
   the cap are recorded and revisitable).  Where no qualifying run exists,
   the single best clean window is used and the degradation is a named
   `no_clean_mile_reasons` entry.
3. **Event overlay wired (audit MML-3).**  `li_series_results` passes mapped
   delay events attached to the series (`sa.delay_events`, e.g. the D6
   event-mapper output) into MML; overlapping windows leave clean-mile
   candidacy, and the overlay's ACTIVE/INACTIVE status is a standing
   disclosure (previously the `events` parameter existed but nothing in the
   pipeline ever supplied it — the exclusion was dead code).
4. **Named no-clean-mile reasons (audit MML-5).**  The boolean conflated
   tight spread with all-windows-evented; `no_clean_mile_reasons` now names
   each condition (tight spread / all windows evented / no sustained run),
   carried into the LI-10 exhibit text.
5. **Disclosures (audit MML-4).**  Standing disclosures state the basis
   policy, run conventions, overlay status, cadence dependence, and the
   preliminary caption (retained).

| No events supplied | silent | overlay INACTIVE disclosed (clean windows unscreened) |
