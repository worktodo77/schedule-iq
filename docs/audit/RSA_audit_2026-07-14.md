## Scope and current behavior

Section 11.5 and LI-15 define RSA as `100 * contested attributed delay / total
attributed delay`, where a window is contested if it is method-sensitive (>2 wd
spread), data-fragile (TRD-05/DAT-05/DUR-04), or path-ambiguous (PCI < 0.20).
The matrix requires each component's classification evidence and an N4
certificate with a responsibility overlay. `analytics/li_provocative.py`
implements `_classify_rsa_windows` (869-909), `rebuttal_surface_area`
(912-956), and `_primary_windows` (959-983).

The scorecard entry is `N20`, `weight: 0`, `surface: internal`, and “needs the
N4 certificate.” RSA therefore cannot affect the grade today, but it can affect
the privileged pre-signature/rebuttal map.

## R1 - `sa=None` is treated as a clean evidence surface (High) - CONFIRMED

When no `SeriesAnalysis` is supplied, `rebuttal_surface_area` sets both
`data_fragile` and `low_pci` to empty sets (around lines 941-946) and proceeds
with certificate-only classification. It does not say that integrity and PCI
checks were unavailable. Every primary window can therefore be labelled
“robust.”

**Probe.**

```text
certificate: overlay=True, one primary window w, total_workdays=10
rebuttal_surface_area(certificate, sa=None):
  rsa_pct = 0
  classification = robust
  evidence = "survives the N4 sweep, integrity checks, and the PCI floor"
  reason = ''
```

The evidence string asserts survival of checks that were not run. This is a
disclosure/integration defect. It can erase exactly the data-fragile and
path-ambiguous components RSA is meant to expose, although the grade remains
unchanged at weight 0.

**Ruling requested:** require the series inputs for a full RSA, return
NOT COMPUTABLE for unavailable integrity/PCI checks, or mark each component's
unavailable dimensions explicitly and never call it robust.

## R2 - One usable N4 variant can yield RSA 0% (Medium) - CONFIRMED

`_primary_windows` compares per-label totals across whatever computable rows
exist, but `rebuttal_surface_area` requires neither a minimum variant count nor
disclosure of skipped rows. With one primary row, no method spread can exist by
definition, so every otherwise-unflagged window is “robust.”

**Probe.** The one-row certificate in R1 returns `rsa_pct=0` and a robust
component, with no reason or “single variant” disclosure. The same output is
also produced if all non-primary N4 rows are non-computable.

This is a methodology/spec gap: RSA's method-sensitive predicate cannot be
tested without a method comparison. It is not a grade defect today, but it can
turn missing N4 coverage into a zero rebuttal surface.

**Ruling requested:** require two or more independent computable variants (or
publish a distinct “method comparison unavailable” state), and retain total,
usable, and skipped variant counts/reasons in the RSA decomposition.

## R3 - Priority classification hides overlapping rebuttal grounds (Medium) - CONFIRMED

`_classify_rsa_windows` applies data-fragile > method-sensitive >
path-ambiguous > robust. The union is correctly counted once, but the output
stores only one `classification` and one evidence string per component. The
other grounds are not named in `RsaComponent`, `class_totals`, or the
interpretation.

**Probe.** A window simultaneously present in all three flag sets returns:

```text
classification = data-fragile
evidence = "window overlaps records failing integrity checks ..."
class_totals = {
  'data-fragile': 8, 'method-sensitive': 0, 'path-ambiguous': 0, 'robust': 0
}
```

The union contribution is 8 workdays (correct), but the method and PCI grounds
are invisible in the privileged evidence artifact. This is a disclosure gap
against the matrix requirement that each component's grounds/evidence be named.

**Ruling requested:** retain a multi-label `grounds`/evidence list while using a
single primary class only for roll-up arithmetic, and add an overlap fixture.

## R4 - Absolute-value normalization changes the signed measure (Medium) - CONFIRMED

`_classify_rsa_windows` sets `delay = abs(float(total_workdays) or 0.0)` (line
885). The matrix formula says attributed delay, but does not authorize replacing
signed movement with absolute movement. A recovery window (negative movement)
is therefore counted as positive delay and can change the denominator and RSA
percentage without a disclosure.

**Probe.**

```text
_classify_rsa_windows([{'window':'w', 'total_workdays':-10}], set(), set(), set())
  total_delay_workdays = 10
  contested_delay_workdays = 0
  rsa_pct = 0
  classification = robust
```

The signed input is not visible in the output. With mixed +10 and -10 windows,
RSA uses a 20-workday absolute denominator even though the signed total is zero.
This is a methodology/spec gap unless N4 explicitly defines RSA over absolute
movement; either way the transformation must be disclosed.

**Ruling requested:** affirm a non-negative attributed-delay domain, or publish
the signed-to-absolute transformation (including mixed-sign handling) and pin
it with fixtures.

## R5 - The N4 responsibility overlay feeding RSA remains code-keyed (High) - CONFIRMED

RSA consumes certificate windows and per-variant allocations, but the upstream
certificate still normalizes responsibility to `code -> party` and allocates
progress/revision by contributor/edit code (`robustness.py` lines 354-370 and
648-675). This violates the UID-first R-ID baseline before RSA classifies the
window.

**Probe.** Same UID, earlier code `OLD-CODE`, later code `NEW-CODE`; the latest
responsibility map contains `NEW-CODE: Owner`, while the contributor record
contains `OLD-CODE`:

```text
progress contributor = {'code': 'OLD-CODE', 'rd_change_workdays': 5.0}
allocated per_party = {'Unallocated': 5.0}
```

The resulting certificate can change its attributed windows and therefore RSA
even though only the display code changed. ARR consumes the same unsafe
certificate allocation.

**Ruling requested:** make N4 responsibility allocation UID-first with a narrow
code fallback and disclosure, then add a re-code RSA/ARR certificate fixture.

## Guardrail confirmation and lower-severity notes

- RSA is privileged/internal, symmetric in stated purpose, and weight 0 in
  `N20` / LI-15.
- The no-certificate and no-overlay NOT COMPUTABLE guards are present.
- The data-fragile priority is deterministic; the finding is that it suppresses
  the other grounds in the evidence surface, not that union arithmetic double
  counts them.

## Proposed tests once rulings land

- R1/R2: missing `SeriesAnalysis`, one usable variant, and skipped variants are
  explicit NOT COMPUTABLE/coverage states.
- R3: overlapping grounds retain all labels/evidence while roll-up counts once.
- R4: signed and mixed-sign windows follow the ruled denominator.
- R5: UID-stable re-code preserves N4 party allocation and RSA inputs.

## Governance package and rulings requested

RSA is weight 0 and privileged, so none of these findings moves a Report Card
grade today. They do affect a discoverable internal rebuttal artifact. Accepted
changes require the governance quartet: methodology/matrix wording,
implementation, seeded fixture/tests, and CHANGELOG note. Promotion to a scored
surface requires a separate spec revision.

1. **R1 - input availability:** do not call unrun integrity/PCI checks robust.
2. **R2 - N4 coverage:** rule the minimum method-variant evidence and disclose
   skipped rows.
3. **R3 - overlap:** preserve all contested grounds/evidence while counting the
   window once.
4. **R4 - signed movement:** rule and disclose signed versus absolute delay.
5. **R5 - identity:** make the certificate responsibility overlay UID-first.
